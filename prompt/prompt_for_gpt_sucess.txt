本專案是：

FPGA-based Super Resolution accelerator implementation。

請先完整理解以下專案背景、設計哲學、目前進度、coding style、verification flow、memory layout、以及使用者偏好。

============================================================
【專案核心目標】
============================================================

目標是：

將輕量化 Single Image Super Resolution (SISR) 模型，
以 hardware-friendly 方式部署到 FPGA。

目前 focus：
- RTL implementation
- fixed-point quantization datapath
- TFLite INT8 deployment flow
- PixelShuffle-based SR reconstruction
- hardware-readable CNN datapath
- golden verified accelerator build flow

不是：
- 高抽象 AI framework
- 高度 reusable software architecture
- 過度 optimization 的 accelerator research prototype

============================================================
【模型架構】
============================================================

目前 SR model：

Input
→ Conv1 (3x3, Cin=1, Cout=8, ReLU)
→ Conv3 (1x1, Cin=8, Cout=4)
→ PixelShuffle x2
→ clipped_relu (0~1)
→ output

scale = 2。

Tensor shape：

input:
(8,8,1)

Conv1 output:
(8,8,8)

Conv3 output:
(8,8,4)

PixelShuffle output:
(16,16,1)

tmp = acc * M0 + M1

rounded = (tmp >>> 32)

if tmp[31] == 1:
    rounded = rounded + 1

q_out = clamp(rounded, -128, 127)

signed 64-bit behavior

先做：

============================================================
【PixelShuffle 設計哲學】
============================================================

目前：
PixelShuffle semantic
與
BRAM/IP integration
必須拆開。

原因：

目前 priority 是：
mapping correctness。

不是：
AXI / BRAM / FSM integration。

============================================================

PixelShuffle 數學：

(8,8,4)
→
(16,16,1)

mapping：

c0 → out[2y  ][2x  ]
c1 → out[2y  ][2x+1]
c2 → out[2y+1][2x  ]
c3 → out[2y+1][2x+1]

PixelShuffle 不改變數值，
只改變位置。

元素總數不變：

8*8*4 = 16*16*1 = 256

============================================================
【目前 Accelerator Build Strategy】
============================================================

採用：

後半段 reconstruction path
先 build。

順序：

1. requant
2. Conv3 MAC
3. Conv3 + requant
4. PixelShuffle
5. 最後才做 Conv1

============================================================

原因：

Conv1 才是真正大魔王。

Conv1 包含：
- line buffer
- padding
- sliding window
- row/col tracking
- streaming
- spatial reuse

因此：
memory-aware architecture
會在 Conv1 才開始重點投入。

============================================================
【PixelShuffle 現階段不要做】
============================================================

不要：
- BRAM IP
- blk_mem_gen
- AXI
- DMA
- output buffer IP
- frame_done FSM
- streaming optimization

目前：
只做 mapping correctness。
