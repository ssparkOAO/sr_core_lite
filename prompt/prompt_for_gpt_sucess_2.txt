============================================================
【更完整的專案背景與設計思維補充】
============================================================

本專案除了做出可運作的 RTL 外，
更重視：

「使用者本人能真正理解 datapath 與硬體設計」。

因此：

請避免：
只追求 code 最短、最 abstract、最 reusable。

本專案更重視：

- 能看懂
- 能 debug
- 能對照數學
- 能對照 tensor
- 能對照 TFLite
- 能理解 memory layout
- 能逐 phase 驗證

============================================================
【使用者目前能力與學習狀態】
============================================================

使用者目前：

已能理解：
- quantization flow
- TFLite INT8 deployment
- zero point / scale
- requantization
- fixed-point datapath
- Conv MAC
- PixelShuffle mapping
- tensor flatten
- NHWC layout
- FPGA RTL datapath
- line buffer 概念
- streaming/window 概念

============================================================

目前使用者最需要的是：

「把 AI 模型數學，
逐步轉成真正 FPGA datapath。」

因此：

請維持：
教學型 datapath RTL。

============================================================
【目前已建立的重要觀念】
============================================================

1.
golden tensor

不是參考而已。

============================================================

2.

都必須：

independent verify。

============================================================

3.

先 functional correctness，

============================================================

4.

往前 build。

============================================================

5.
目前 architecture philosophy：

semantic correctness
>
throughput optimization。

============================================================

1. code readability

============================================================
【目前已建立的硬體設計風格】
============================================================

目前所有 RTL：

傾向：

「像 block diagram 的 code」。

============================================================

例如：

input
→ multiplier
→ partial sum
→ accumulator
→ requant
→ clamp

============================================================

使用者偏好：

訊號名稱明確。

例如：

mul00
psum0
acc0

而不是：

大量抽象化 generate architecture。

============================================================
【目前不喜歡的風格】
============================================================

避免：

- 超高抽象 reusable framework
- 大量 nested generate
- systolic array
- PE mesh
- overly smart architecture
- accelerator research prototype style
- 過度 optimization 導致可讀性下降

============================================================
【目前專案的真正目標】
============================================================

不是：

「做 benchmark paper accelerator」。

而是：

「做一個：
自己完全理解、
完全能 debug、
完全能驗證、
可部署 FPGA 的 SR accelerator。」

============================================================

目前 priority：

1. correctness
2. readability

4. hardware intuition
