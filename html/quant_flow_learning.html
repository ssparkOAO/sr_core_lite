1. 這次學到的核心觀念

int8 TFLite 部署到硬體時，不是每一層都還原成 float，再重新量化。 正確概念是：每一層 Conv 先產生 int32 accumulation，接著做 requantization， 把 int32 轉成下一層要吃的 int8 activation。 

uint8 image
  → input preprocessing / quantization
  → int8 activation
  → Conv MAC: int8 input * int8 weight
  → int32 accumulation
  → add int32 bias
  → requantization: int32 → int8
  → next layer input int8

所以「量化還原」比較容易讓人誤會。硬體裡真正需要的是 requantization，也就是把一層的 int32 計算結果轉回另一個 int8 scale。

2. Bias 不是 int8，而是 int32

已確認 ref/cnn_classfication/CNN_training/models/weights.pkl： Conv 和 Dense 的 weight 是 int8，bias 是 int32。

資料 dtype 原因
input activation int8 或模型輸入可能是 uint8 節省資料量，符合 TFLite quantized tensor。
weight int8 Conv 使用 int8 乘法。
MAC accumulation int32 很多個 int8 乘積相加，必須用更寬的位元避免溢位。
bias int32 bias 要加在 int32 accumulation 尺度，因此不是 int8。
layer output int8 經過 requantization 後，提供給下一層。

Bias 的 scale 通常是： 

bias_scale = input_scale * weight_scale

因此 bias 放在 int32 accumulation domain 裡最合理。

3. 一層 Conv 的量化公式

硬體 MAC 常先算： 

acc = sum(q_input * q_weight)

但 TFLite 的真實數學關係是： 

real_input  = input_scale  * (q_input - input_zero_point)
real_weight = weight_scale * q_weight
real_output = output_scale * (q_output - output_zero_point)

所以 requantization 要做： 

q_output =
  round((acc - input_zero_point * sum(weight) + bias)
        * input_scale * weight_scale / output_scale)
  + output_zero_point

為了讓 RTL 不使用浮點數，可以預先產生 M0 和 M1： 

M0 = floor((input_scale * weight_scale / output_scale) * 2^32)
M1 = M0 * (bias - input_zero_point * sum(weight)) + (output_zero_point << 32)

tmp   = M0 * acc + M1
q_out = round(tmp / 2^32)
q_out = clamp(q_out, -128, 127)

4. CNN Classification 的 PS/PL 流程

cnn_classfication 參考專案裡要分成兩種理解方式。

4.1 Python / PS 分層控制版本

在 CNN_training/library/inference_hardware.py 的流程裡， PL 主要負責算 Conv / FC 的 int32 accumulation，PS/Python 再做 requantization。 

PS:
  image resize / grayscale
  uint8 image → int8 input，例如 image - 128
  DMA send int8 activation

PL:
  讀 int8 activation
  讀 int8 weight
  做 MAC
  回傳 int32 accumulation

PS:
  DMA recv int32 accumulation
  用 M0/M1 做 requantization
  int32 → int8
  把 int8 output 當下一層 input
  再啟動下一層 PL 計算

這種方式的優點是容易驗證。缺點是每一層都要 PS/PL 來回一次，效能比較差。

4.2 完整 RTL Network 版本

在 LAB/Lab_final/.../new 裡，可以看到某些 RTL module 已經把 requantization 和 clamp 寫在 PL 內，例如 ConV1.v、FC_2.v。 

PL:
  Conv MAC int32
  M0/M1 fixed-point requantization
  clamp to int8
  寫入內部 activation buffer
  下一層直接讀 int8 activation

這種方式比較接近最終部署，但 RTL 複雜度較高，也比較不好第一時間 debug。

5. 對 SR Core 的設計啟發

對目前的 Super Resolution 小模型來說，建議採取逐步策略。

階段 做法 目的
驗證階段 PL 可先只輸出 int32 acc，PS/Python 做 requant。 確認 MAC、window、weight 排列、zero point 修正都正確。
Core 完整階段 把 int32 bias、M0/M1 requant、clamp 放進 PL。 讓 Conv1 output 直接成為 Conv3 input。
模型整合階段 Conv1 → ReLU → Conv3 → PixelShuffle 全部在 PL pipeline 裡串起來。 減少 PS/PL 來回，接近最終 FPGA 推論。

最終 SR core 內部資料流應該是： 

input int8
  → Conv3x3 MAC int32
  → add int32 bias
  → M0/M1 requant int32 to int8
  → ReLU / clamp
  → Conv1x1 MAC int32
  → add int32 bias
  → M0/M1 requant int32 to int8
  → PixelShuffle
  → output int8 / uint8

6. 本專案目前採用的學習路線

先用 PS/Python 做 requantization 來學清楚數學與資料流，再把 requantization 搬進 PL。 這樣比較容易知道錯誤是在 MAC、bias、scale、rounding、clamp，還是 PixelShuffle mapping。 

Step 1: Python extractor 產生 weight / bias / M0 / M1
Step 2: Python golden 產生每層 intermediate tensor
Step 3: RTL 先驗證單層 MAC
Step 4: RTL 加入 bias + requant
Step 5: RTL 串接 Conv1 / Conv3 / PixelShuffle
