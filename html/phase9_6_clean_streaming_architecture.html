Phase9.6 Clean Streaming CNN / RAM Architecture

文件位置：model_lite/sr_core/html/phase9_6_clean_streaming_architecture.html

RTL_sys：model_lite/sr_core/RTL_sys/phase9_6_clean_stream/

Vivado Project：model_lite/sr_core/vivado/sr_core_clean_stream_pynqz2/

本階段目標不是重新設計 CNN 演算法，而是把 Phase9.5 已經 PASS 的 image-level flow 整理成更像實際 PL datapath 的乾淨 hierarchy：Conv1、Conv3、ROM IP、Output RAM IP 保持同層可見，Conv3 output 直接 streaming 到 PixelShuffle。

1. Phase9.5 遇到的架構問題

問題 影響 Phase9.6 修正
Conv1 block 有 win00_dbg 系列 output 這些 window debug 線在 image top 沒使用，會讓 schematic 和 port list 變吵。 新的 sr_conv1_3x3_cin1_cout8_flat 不再輸出 window debug port。
Conv1 / Conv3 parameter ports 過度展開 top 需要連接大量 w000、m0_0、m1_0 類 port。 改成分組 flat bus，保留平行運算但減少 top-level wiring。
Conv3 feature RAM 不必要 Conv3 是 1x1，同一 pixel 的 4-channel output 可以直接送 PixelShuffle。 移除 conv3_feature_ram，Conv3 output 直接 streaming 到 PixelShuffle。
top-level dbg_* port 太多 debug readback 介面混在正式 top interface 裡，不像最後 PL module。 改成正式 output RAM readout：out_rd_en/out_rd_addr/out_rd_data。

2. 新資料流

TB input source
  -> sr_conv1_3x3_cin1_cout8_flat
  -> sr_conv1_to_conv3_stream_slice
  -> sr_conv1x1_cin8_cout4_flat
  -> pixel_shuffle_core
  -> sr_output_pack2x2_uint8
  -> output_image_ram

這個 flow 保留 output image RAM，因為最後影像需要被外部讀回；但不再保留 Conv3 feature RAM， 因為那只是 Phase9.5 為了拆階段 debug 而留下的中間 buffer。

3. Flat Bus 不是序列化

Flat bus 只是把多個同類參數包成一條寬 bus，例如 Conv1 weight 是 conv1_weight_flat[575:0]。block 內部仍拆成 signed wire 後同時接到 MAC， 所以 parallel MAC datapath 沒有被改成 time-multiplex，也沒有犧牲平行化。

Layer Flat bus 內容
Conv1 3x3 conv1_weight_flat[575:0] 72 個 signed int8 weight
Conv1 requant conv1_m0_flat[255:0], conv1_m1_flat[511:0] 8 組 M0/M1
Conv3 1x1 conv3_weight_flat[255:0] 32 個 signed int8 weight
Conv3 requant conv3_m0_flat[127:0], conv3_m1_flat[255:0] 4 組 M0/M1

4. Vivado Project

新 project 使用功能導向命名：sr_core_clean_stream_pynqz2。 檔名不使用 p96 這種無語意縮寫，保留 kernel、Cin、Cout 等資訊。 

vivado/sr_core_clean_stream_pynqz2/
  src/rtl/
  src/tb/
  src/manifest/SOURCE_SNAPSHOT_MAP.txt
  tcl/run_phase9_6_clean_stream_sim.tcl

此 project 會建立 6 個 parameter ROM IP 與 1 個 output_image_ram。 不建立 conv3_feature_ram。

5. 驗證方式

1. TB reads butterflyx2_Y.txt
2. TB converts uint8 input to signed int8 by subtracting 128
3. PL RTL runs Conv1 -> Conv3 -> PixelShuffle -> OutputStage
4. PL writes packed uint8 output into output_image_ram
5. TB reads output_image_ram through formal out_rd_* port
6. TB writes sr_output_uint8_clean_stream.txt
7. TB compares with Phase9.5 sr_output_uint8.txt

本階段不再產生 PNG，也不計算 PSNR。Phase9.6 的目標是確認乾淨 streaming architecture 與 Phase9.5 已驗證輸出完全一致。

6. PASS 條件

input sent count = 16384
output pixel count = 65536
mismatch count = 0
max abs diff = 0
PASS

比對基準是 pic/test_pic/result/sr_output_uint8.txt， 輸出檔是 pic/test_pic/result/sr_output_uint8_clean_stream.txt。

7. What Did We Learn?

Phase9.6 釐清了「介面乾淨」和「運算平行」不是衝突的。參數可以用 flat bus 收斂 port 數，但 CNN block 內仍保持清楚的 parallel datapath。中間 feature RAM 只有在需要跨階段保存或外部讀寫時才合理；Conv3 到 PixelShuffle 這段可以直接 streaming。
