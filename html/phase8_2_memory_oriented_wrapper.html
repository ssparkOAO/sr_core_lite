Phase8.2 Memory-Oriented Top Wrapper

文件位置：model_lite/sr_core/html/phase8_2_memory_oriented_wrapper.html

返回：Phase8 完整紀錄

Phase8.2 建立 sr_core_top_mem_wrapper.v。 這個 wrapper 將 weight/M0/M1 收進內部 memory-style arrays， 再轉接到已 verified 的 sr_core_top。

1. 為什麼需要 Wrapper

原本的 sr_core_top 是 verification-oriented top。 它把 Conv1/Conv3 weight 和 M0/M1 全部攤成 top-level input ports， 這對早期驗證很方便，但不適合 accelerator architecture。 

舊方式：
TB -> many weight/M0/M1 ports -> sr_core_top

Phase8.2：
TB -> wrapper internal memories -> verified sr_core_top -> output_uint8

2. 為什麼不直接修改 Verified RTL

model_lite/sr_core/RTL 目前是 Golden RTL Version。 若直接修改 sr_core_top，Phase7 的 PASS baseline 就不再是穩定參考。 所以 Phase8.2 把新架構放在 RTL_sys/phase8_weight_memory/。

這個 wrapper 是外層 architecture refactor，不是 compute datapath rewrite。 Conv1、Conv3、PixelShuffle、OutputStage、requantize 都仍然使用已 verified modules。

3. Wrapper 和 sr_core_top 差異

項目 sr_core_top sr_core_top_mem_wrapper
定位 Verification-oriented top Memory-oriented architecture wrapper
Weight/M0/M1 大量 top-level input ports 內部 memory-style arrays
Compute datapath 已 verified 沿用已 verified sr_core_top
未來相容性 適合 simulation verification 更接近 ROM/BRAM/AXI parameter storage

4. 使用的 Memory

conv1_weight_mem [0:71]  signed int8
conv3_weight_mem [0:31]  signed int8
conv1_m0_mem     [0:7]   signed int32
conv1_m1_mem     [0:7]   signed int64
conv3_m0_mem     [0:3]   signed int32
conv3_m1_mem     [0:3]   signed int64

Phase8.2 目前仍由 TB 透過 hierarchical assignment 載入 memory。 這是中間驗證版本，尚未加入 BRAM IP、AXI、DMA 或 weight loader FSM。

6. 驗證流程

TB
  -> load weight/M0/M1 files into wrapper memories
  -> load input_int8_for_core.mem
  -> run sr_core_top_mem_wrapper
  -> wrapper connects memory values to verified sr_core_top
  -> compare output_mem with output_uint8.mem
  -> dump feature_mem0/1/2 for layer-by-layer debug

7. 設計取捨

方案 是否採用 原因
直接修改 verified sr_core_top 否 會破壞 Golden RTL baseline。
直接導入 BRAM IP 否 太早引入 read latency、XCI、初始化格式問題。
建立 memory-oriented wrapper 是 降低 top-level ports，同時維持目前 PASS 行為。
