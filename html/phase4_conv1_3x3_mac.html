本頁記錄 Phase4：Conv1 的 3x3 window generation + MAC。 目前只驗證 input_int8_for_core -> conv1_acc_int32， 不做 requant、ReLU、Conv3 或 PixelShuffle。

1. 本階段目標

input_int8_for_core.mem
  -> sr_window_3x3_cin1
  -> sr_conv3x3_cin1_cout8_mac
  -> conv1_acc_int32.mem

Conv1 是模型中的第一個 convolution layer，kernel 是 3x3， Cin=1，Cout=8。

2. Tensor Shape

Tensor Shape 說明
input_int8_for_core.mem (8,8,1) Conv1 input，signed int8。
conv1_weight.mem [8][3][3][1] Cin=1，所以可視為 weight[co][ky][kx]。
conv1_acc_int32.mem (8,8,8) Conv1 MAC golden，signed int32。

3. Padding 為什麼是 PAD_VALUE

Conv1 input 是 quantized int8 tensor。影像外圍 padding 不能用數學上的 0， 而要使用 input zero point。此模型的 input zero point 是： 

PAD_VALUE = -128

如果 padding 錯用 0，Conv1 border 位置的 accumulator 會明顯 mismatch。 所以 RTL 裡把 padding 做成 architecture-level parameter， 避免 -128 散落在各處。

4. Window Concept

sr_window_3x3_cin1 負責在 RTL 內產生 padded 3x3 window。 Testbench 不幫 DUT 做 window，也不幫 DUT 補 padding。 

win00 win01 win02
win10 win11 win12
win20 win21 win22

目前 Phase4 已改成 streaming line-buffer architecture。 Window core 內部走一個 padded raster： (IMG_H+2) x (IMG_W+2)。遇到 padding 位置時， RTL 自己注入 PAD_VALUE；遇到真實影像位置時， 用 in_ready 向外部 input source 要下一個 signed int8 pixel。

5. Line Buffer Concept

Streaming 版本使用 two line buffers 保存前兩列 padded-stream 資料， 再配合目前 padded-stream pixel 形成 3x3 window： 

two rows up   -> win00 win01 win02
previous row  -> win10 win11 win12
current row   -> win20 win21 win22

這版不是 frame memory random access。Pixel stream 一邊進， line buffer 一邊更新，3x3 window 一邊滑動產生。 Padding row / padding column 也走同一條 padded-stream datapath， 所以 top、left、right、bottom padding 都由 RTL window core 處理。

6. x/y Tracking 與 window_valid

Window module 會輸出 out_x、out_y 表示目前 window 對應的 output coordinate。

當 window_valid=1 時，win00~win22 對應該座標的 padded 3x3 window，MAC output 可與 conv1_acc_int32.mem 的同一個 pixel index 比對。

in_ready 只會在 padded raster 的真實影像位置拉高。 Testbench 模擬外部 pixel source：只有看到 in_ready 才送下一個 input_int8_for_core.mem pixel。 因此 TB 沒有替 DUT 做 padding 或 window 排列。

7. 3x3 MAC 數學

acc[co] =
    sum ky=0..2
    sum kx=0..2
        window[ky][kx] * weight[co][ky][kx]

目前 MAC 只做 signed int8 x signed int8 的 multiply accumulate， output 是 signed int32 accumulator。不加 bias、不做 requant、不做 ReLU。

8. Weight Mapping

conv1_weight.mem flatten 後是 output-channel major。 每個 output channel 有 9 個 3x3 weights： 

co0: w000~w008
co1: w100~w108
co2: w200~w208
...
co7: w700~w708

9. Golden Verification Flow

1. 讀 input_int8_for_core.mem
2. 讀 conv1_weight.mem
3. 讀 conv1_acc_int32.mem
4. input stream follows in_ready -> sr_window_3x3_cin1
5. window_valid -> sr_conv3x3_cin1_cout8_mac
6. compare acc0~acc7 with conv1_acc_int32.mem

若 mismatch，testbench 會印出 x、y、 output channel、RTL acc、golden acc，以及完整 3x3 window。 這對 debug padding 和 window mapping 很重要。

10. PASS 條件

mismatch count = 0
max abs diff   = 0
