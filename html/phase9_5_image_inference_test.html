Phase9.5 Image-Level SR Inference Test

文件位置：model_lite/sr_core/html/phase9_5_image_inference_test.html

Workspace：model_lite/sr_core/RTL_sys/phase9_5_image_test/

Vivado Project：model_lite/sr_core/vivado/sr_core_streaming_pynqz2/sr_core_streaming_pynqz2.xpr

Phase9.5 的目標是把測試從 8x8 小圖 golden verification 推進到真正影像等級： testbench 只負責讀取 LR 影像文字檔並做 uint8 - 128 的 input formatting， Conv1、Conv3、PixelShuffle、OutputStage 與 RAM buffer 都由 PL RTL / Vivado BMG IP 完成。

1. 為什麼 Phase9.4 不夠

Phase9.4 已經證明 reference-like hierarchy 可以使用 Vivado BMG ROM/RAM IP， 並且在 8x8 LR patch 上得到 bit-exact PASS。可是 8x8 top 的 counter、address width、 RAM depth 都是小圖尺寸，因此不能直接拿來處理 128x128 LR 影像。

項目 Phase9.4 小圖 Phase9.5 影像測試
LR input 8x8 = 64 pixels 128x128 = 16384 pixels
HR output 16x16 = 256 pixels 256x256 = 65536 pixels
Conv3 feature RAM 256 entries of int8 16384 entries of packed 4-channel int8
Output RAM 128 words when 16-bit packed 32768 words when 16-bit packed

2. Phase9.5 架構

TB image source
  -> read butterflyx2_Y.txt
  -> uint8 - 128
  -> sr_nn_top_img
       -> parameter ROM IP
       -> preload parameter register bank
       -> Conv1 Block, IMG_W=128, IMG_H=128
       -> Conv1 to Conv3 register slice
       -> Conv3 Block
       -> conv3_feature_ram_img
       -> PixelShuffle x2
       -> OutputStage packer
       -> output_image_ram_img
  -> TB debug readback
  -> sr_output_uint8.txt
  -> Python image evaluation

這裡仍然不做 AXI、DMA、PS-PL，也不做 camera input。輸入圖片來源先由 testbench 模擬， 但輸入進入 DUT 之後的 CNN 與影像輸出格式轉換都留在 PL RTL 中。

3. 新增 RTL 與 Testbench

檔案 用途
sr_nn_top_img.v 128x128 image-size top，讓 parameter ROM、Conv block、RAM IP、PixelShuffle 同層可見。
sr_ctrl_img.v 控制 16384 個 LR pixels，寫入 Conv3 feature RAM，再啟動 PixelShuffle phase。
sr_c1c3_slice_img.v Conv1 to Conv3 register slice，保留同一 pixel 的 8-channel output，不建立 feature_mem0。
sr_out_pack_img.v 包住四個 sr_output_stage，將 PixelShuffle 的 2x2 int8 output 轉成 uint8 packed words。
tb_sr_nn_top_img.v 讀入影像文字檔，餵入 DUT，完成後從 output RAM debug port 讀回並輸出 SR text。

4. Vivado BMG IP 規劃

Phase9.5 使用同一個 streaming Vivado project，不新建 project，也不覆蓋 Phase9.4 小圖 flow。 參數仍使用既有 6 個 ROM IP，影像尺寸資料另外新增 image-size RAM IP。

IP Memory Type Width Depth 原因
conv3_feature_ram_img Simple Dual Port RAM 32 16384 每個 LR pixel 存 4 個 int8 Conv3 output，剛好 packed 成 32-bit word。
output_image_ram_img True Dual Port RAM 16 32768 PixelShuffle 每次產生左右相鄰兩個 uint8 pixel，16-bit word 可直接對應水平 pair。

Vivado GUI 對應設定

GUI Option conv3_feature_ram_img output_image_ram_img 為什麼
IP Catalog Block Memory Generator
Memory Type Simple Dual Port RAM True Dual Port RAM Conv3 RAM 一寫一讀即可；output RAM 需要兩個 write port 對應 PixelShuffle even/odd row。
Write Width / Read Width 32 16 配合資料 packing，減少 address 數量並讓 schematic 更接近 memory-port architecture。
Write Depth 16384 32768 128x128 LR 與 256x256 HR packed pair 所需容量。
Enable Pin Use ENA / ENB Pin Use ENA / ENB Pin 控制器明確控制 read/write transaction，方便波形 debug。
Output Register Disabled Disabled 維持 BMG primitive 1-cycle read latency，避免多一拍造成 controller 更複雜。

5. Testbench Preprocessing

butterflyx2_Y.txt 是 128x128、每行一個 hex uint8。TB 讀取後只做一件事： 將 uint8 image domain 轉回模型 core 使用的 signed int8 domain。 

input_uint8 0..255
  -> input_int8_for_core = input_uint8 - 128
  -> feed sr_nn_top_img

這個動作是 input formatting，不是 CNN 計算。Conv1、Conv3、PixelShuffle 與 final uint8 conversion 都在 DUT 內完成。

6. Python Evaluation Tool

tools/sr_image_eval.py 負責將 TB 輸出的 sr_output_uint8.txt 轉成 PNG，產生 bilinear x2 baseline，並與 HR 圖計算 PSNR。同時它也會呼叫既有 quantized golden flow， 驗證 RTL SR output 是否和 software golden bit-exact。 

python tools/sr_image_eval.py

輸出檔案：

  • pic/test_pic/result/sr_output.png
  • pic/test_pic/result/bilinear_x2.png
  • pic/test_pic/result/software_golden_output.png
  • pic/test_pic/result/image_eval_report.md
  • pic/test_pic/result/image_eval_summary.txt

7. 驗證流程

1. Confirm input image size: 128x128 grayscale
2. Confirm HR image size: 256x256 grayscale
3. Run Vivado Phase9.5 image simulation
4. TB writes sr_output_uint8.txt
5. Run Python image evaluation
6. Compare RTL output with software golden
7. Compute PSNR(SR, HR) and PSNR(Bilinear, HR)

Vivado Tcl：

9. 目前遇到的問題

本階段一開始遇到兩個容易誤判的問題。第一個是 Vivado GUI 開著同一個 project 時， batch Tcl 只能用 read-only 方式開啟 project。第二個是 shell sandbox 不能讓 Vivado 寫入 project， 因此需要用允許寫入的方式執行 Vivado batch。

解法是先關閉 GUI 中的 streaming project，接著執行 run_phase9_5_image_sim.tcl。完成後 Vivado 成功建立 conv3_feature_ram_img 與 output_image_ram_img， 並完成 image-level simulation。

10. What Did We Learn?

Phase9.5 的重點不是只把小圖測試放大，而是確認 PL-side CNN + RAM buffer architecture 可以承接真實影像尺寸。 128x128 會立刻暴露 address width、RAM depth、counter range、PixelShuffle HR address 等問題， 因此它是進入更完整 PL integration 前很重要的一關。

這一階段也釐清了 Vivado 專案管理：同一個 `.xpr` 不能同時由 GUI 與 batch Tcl 以可寫方式操作。 若要自動建立 IP、更新 source 或跑 simulation，必須確保 project 沒有被另一個 Vivado session 持有。
