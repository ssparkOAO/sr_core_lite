Phase10.1 AXI-Stream Wrapper Debug and Verification

文件位置：model_lite/sr_core/html/phase10_ps_dma_wrapper_plan.html

RTL 位置：model_lite/sr_core/RTL_sys/phase10_ps_dma_wrapper/

本階段目標是把 Phase9.6 clean-stream SR core 包成 AXI-Stream style wrapper， 並確認 wrapper control、input handshake、output handshake、value compare 都正確。

1. 先固定 Phase9.6 Baseline

這次 debug 的第一步不是修改 CNN datapath，而是先確認 Phase9.6 clean-stream baseline 是否可信。 原因是本地曾出現一個 stale result txt 顯示 FAIL，但 Vivado clean-stream project 的最新 simulate.log 顯示 PASS，而且 clean-stream output 與 Phase9.5 reference hash 完全相同。

結論：Phase9.6 clean-stream core 目前視為 golden wrapper reference。 不應再用 stale 的 tb_sr_top_clean_stream_img_result.txt 判斷 CNN arithmetic 是否壞掉。

檢查項目 結果
Phase9.6 Vivado simulate.log mismatch count = 0、max abs diff = 0、PASS
Clean-stream txt vs Phase9.5 txt sr_output_uint8_clean_stream.txt 與 sr_output_uint8.txt hash 相同
修正動作 將 Phase9.6 stale result txt 更新為最新 PASS 狀態

2. 真正 Root Cause

問題不是 Conv1、Conv3、PixelShuffle 或 OutputStage 的 arithmetic 先壞掉。 這次主要混在一起的問題有兩個：

問題 影響 修正方式
stale / 被覆寫的 Phase9.6 result txt 看起來像 clean-stream core FAIL，但實際最新 Vivado log 是 PASS。 以最新可信 replay 與 output hash 作為 baseline。
手動 xsim flow 使用 *_sim_netlist.v 容易和 Phase9.6 PASS flow 使用的 BMG behavioral wrapper 不一致，造成假 mismatch。 Phase10 wrapper sim 改用 Vivado IP 的 behavioral simulation wrapper。
wrapper sequencing 不夠清楚 可能在 core 尚未 ready 時打開 s_axis_tready，造成 timeout 或 input/control 不同步。 加入 WAIT_CORE_READY 狀態，input handshake 只在 core ready 後發生。

3. Wrapper FSM

Phase10 wrapper 現在使用明確分段的 control flow。這樣 debug 時可以先看 control 是否完整， 再看 value compare，而不是把 timeout 和數值 mismatch 混在一起。 

PRELOAD
  -> START_CORE
  -> WAIT_CORE_READY
  -> STREAM_INPUT
  -> WAIT_CORE_DONE
  -> READ_OUTPUT
  -> SEND_AXIS
  -> DONE
狀態 責任
PRELOAD等待 Phase9.6 core 內部 parameter ROM preload 完成。
START_CORE送出 one-cycle core start pulse。
WAIT_CORE_READY等待 clean-stream core 的 input side ready。
STREAM_INPUT接收 16384 筆 LR uint8 input，wrapper 內部轉成 signed int8。
WAIT_CORE_DONE等待完整 Conv1、Conv3、PixelShuffle、OutputStage 完成。
READ_OUTPUT透過正式 output RAM readout port 讀回 256x256 uint8 output。
SEND_AXIS輸出 65536 筆 M_AXIS uint8 pixel，最後一筆拉高 m_axis_tlast。
DONE拉高 wrapper done，結束測試。

4. Simulation Model 規則

Phase10 wrapper verification 必須使用和 Phase9.6 PASS flow 一致的 Vivado BMG behavioral simulation wrapper。 不建議在這個層級直接改用 *_sim_netlist.v 當作主要判斷來源，因為那會把 IP model 差異與 wrapper control debug 混在一起。 

Use:
  vivado/sr_core_clean_stream_pynqz2/.../ip/*/sim/*.v
  one blk_mem_gen_v8_4.v behavioral model

Avoid for this verification:
  *_sim_netlist.v as the primary source

新增的 helper script 位置： 

RTL_sys/phase10_ps_dma_wrapper/sim/run_phase10_axis_wrapper_sim.ps1

執行方式： 

powershell -ExecutionPolicy Bypass -File .\run_phase10_axis_wrapper_sim.ps1

5. Testbench 判斷方式

TB 不再只輸出一個 mismatch count。現在會把 verification 拆成四層，方便定位錯誤屬於 control、handshake，還是真正數值 mismatch。

檢查層 PASS 條件
Input handshake input count = 16384，input tlast index = 16383
Core done core_done = 1，wrapper error = 0
Output handshake output count = 65536，output tlast index = 65535
Value compare mismatch count = 0，max abs diff = 0

7. 下一步

Phase10.1 已證明 AXI-Stream shell 可以正確接住 Phase9.6 clean-stream SR core。 下一步可以開始整理更完整的 Vivado block design / AXI DMA / PS-PL integration，但仍應保留目前這個 wrapper simulation 作為 regression checkpoint。

  • 保留 Phase9.6 clean-stream project 作為 PL-side CNN/RAM golden baseline。
  • 保留 Phase10 wrapper sim 作為 AXIS wrapper regression。
  • 若 Vivado batch 顯示 project read-only，先關閉 Vivado GUI，再重新跑 Tcl。
  • 後續若切到 block design，仍要先比對 txt equivalence，再進一步做 image/PSNR 評估。
