Phase9.4 BMG IP-Backed Verification

文件位置：model_lite/sr_core/html/phase9_4_bmg_ip_backed_verification.html

Vivado Project：model_lite/sr_core/vivado/sr_core_streaming_pynqz2/sr_core_streaming_pynqz2.xpr

IP-backed TB：model_lite/sr_core/RTL_sys/phase9_4_ip_verify/sim/tb_sr_nn_top_ip.v

Phase9.4 的目標是確認 Phase9.3 的 sr_nn_top 不只可以用 behavioral RAM model 跑通，也可以接 Vivado Block Memory Generator 產生的 ROM/RAM IP simulation wrapper，並完成 end-to-end golden verification。

1. Phase9.3 和 Phase9.4 的差異

項目 Phase9.3 Phase9.4
Memory source ram_ip_bhv.v 手寫 behavioral model Vivado project 內 BMG IP wrapper
測試目的 確認 hierarchy 與 controller 行為正確 確認真正 IP-backed simulation 仍然 bit-exact
TB 位置 RTL_sys/phase9_3_ref_hier/sim/tb_sr_nn_top.v RTL_sys/phase9_4_ip_verify/sim/tb_sr_nn_top_ip.v
結果位置 RTL_sys/phase9_3_ref_hier/results/ RTL_sys/phase9_4_ip_verify/results/

2. 真正用到的 Vivado IP

Phase9.4 使用既有 Vivado project，不重新建立 project，也不重建 IP。 目前 project 內真正需要的 memory IP 是：

  • conv1_weight_rom
  • conv3_weight_rom
  • conv1_m0_rom
  • conv1_m1_rom
  • conv3_m0_rom
  • conv3_m1_rom
  • conv3_feature_ram
  • output_image_ram

Vivado compile 時會使用 project 產生的 wrapper，例如： 

.../ip/conv1_weight_rom/sim/conv1_weight_rom.v
.../ip/conv3_feature_ram/sim/conv3_feature_ram.v
.../ip/output_image_ram/sim/output_image_ram.v

目前 project 已設定為 target_language = Verilog、 simulator_language = Verilog。實際 simulation log 中， Vivado 使用 xvlog 分析上述 .v wrapper。 Project compile-order 裡沒有 .vhd 或 .vhdl source。

3. 為什麼需要 Debug Readback Port

Phase9.3 的手寫 RAM model 有公開的 mem[] array， testbench 可以直接偷看記憶體內容。但 Vivado BMG IP wrapper 不是這樣使用的； testbench 不應依賴 IP 內部實作細節。

因此 Phase9.4 在 sr_nn_top 加入 debug readback ports。 正常 CNN run 時，RAM port 由 controller 控制；等 done=1 後， TB 才透過 debug port 讀回 RAM 內容並比對 golden。 

dbg_c3_rd_en / dbg_c3_rd_addr / dbg_c3_rd_data
dbg_out_rd_en / dbg_out_rd_addr / dbg_out_rd_data

4. 驗證流程

Vivado BMG ROM/RAM IP
→ sr_nn_top
→ preload parameter
→ feed input_int8_for_core.mem
→ Conv1
→ Conv1-to-Conv3 register slice
→ Conv3
→ conv3_feature_ram IP
→ PixelShuffle
→ OutputStage packer
→ output_image_ram IP
→ debug readback
→ compare golden

使用 golden：

  • generated/golden/input_int8_for_core.mem
  • generated/golden/conv3_out_int8.mem
  • generated/golden/output_uint8.mem

5. PASS 條件與結果

conv3_feature_ram mismatch count = 0
output_image_ram mismatch count = 0
total mismatch count = 0
max abs diff = 0
PASS

這代表目前 sr_nn_top 的 reference-like hierarchy 可以接上 Vivado BMG ROM/RAM IP simulation wrapper，而且結果與 golden bit-exact。

6. 操作指令

Phase9.4 的主要驗證入口是： 

C:\Xilinx\Vivado\2023.1\bin\vivado.bat -mode batch -source model_lite/sr_core/vivado/sr_core_streaming_pynqz2/tcl/run_phase9_4_ip_sim.tcl

如果 Vivado project 來源清單變亂，可以先執行清理腳本： 

C:\Xilinx\Vivado\2023.1\bin\vivado.bat -mode batch -source model_lite/sr_core/vivado/sr_core_streaming_pynqz2/tcl/clean_phase9_4_project.tcl

如果要重新確認 IP wrapper 使用 Verilog，執行： 

C:\Xilinx\Vivado\2023.1\bin\vivado.bat -mode batch -source model_lite/sr_core/vivado/sr_core_streaming_pynqz2/tcl/set_ip_verilog_and_regen.tcl
C:\Xilinx\Vivado\2023.1\bin\vivado.bat -mode batch -source model_lite/sr_core/vivado/sr_core_streaming_pynqz2/tcl/report_ip_language.tcl

7. Warning Notes

訊息 說明 處理方式
BMG is using a behavioral model for simulation 這是 Vivado BMG IP 自己的 simulation model 提醒，不是使用 ram_ip_bhv.v。 正常保留；功能驗證仍以 golden compare 為準。
No write access right to the local Tcl store Vivado 環境找不到可寫入的 Tcl store。 與 RTL/IP correctness 無關；目前不影響 simulation PASS。
Unable to auto find GCC executables XSim 找不到 GCC path。 本專案沒有使用 DPI/C simulation，不影響 PASS。

8. What Did We Learn?

Phase9.3 是 hierarchy correctness，Phase9.4 是 IP-backed correctness。 兩者都很重要，但不能混在一起看。從現在開始，若要確認「真的用 Vivado BMG IP」， 請看 Phase9.4 的 Vivado project、IP wrapper、與 phase9_4_ip_verify/results/。
