Phase7 SR Core Top

紀錄位置：model_lite/sr_core/html/phase7_sr_core_top.html

RTL：model_lite/sr_core/RTL/sr_core_top/

回到主頁：SR Core 專案首頁

這一階段建立的是 Verification-Oriented Top，目的不是部署到 FPGA， 而是把已經獨立 verified 的 Conv1、Conv3、PixelShuffle、OutputStage 串起來， 做一次完整 end-to-end golden verification。

1. 整體 Accelerator Flow

input_int8_for_core
  -> Conv1 3x3 block
  -> feature_mem0
  -> Conv3 1x1 block
  -> feature_mem1
  -> PixelShuffle x2 transaction core
  -> feature_mem2
  -> OutputStage
  -> output_uint8

目前每一個 block 都已經獨立和 runtime-verified golden 比對通過。 Phase7 的重點是確認：把它們串起來後，完整輸出仍然等於 generated/golden/output_uint8.mem。

2. 為什麼先做 Verification-Oriented Top

這個 top 不是最終部署版，所以刻意不加入 AXI、DMA、BRAM IP、weight loader、 memory controller 或 throughput optimization。這樣做可以讓目前問題單純保持在： RTL arithmetic、tensor mapping、stage ordering 是否完全正確。

目前策略是先證明 compute pipeline 正確，再逐步替換 memory bridge。 這比一開始就加入外部 memory subsystem 更容易定位錯誤。

3. Stage Memory Bridge

sr_core_top 內部使用 behavioral memory 做 stage bridge。 每個 major stage 完成後，把結果先寫入一個清楚的 feature memory， 下一個 stage 再從該 memory 讀資料。

Memory 內容 Golden 對照
feature_mem0 Conv1 full block output，shape = (8,8,8) conv1_out_int8.mem
feature_mem1 Conv3 1x1 block output，shape = (8,8,4) conv3_out_int8.mem
feature_mem2 PixelShuffle output，shape = (16,16,1) pixel_shuffle_out_int8.mem

這些 memory 不是 BRAM IP，只是 verification 階段使用的 behavioral memory。 好處是每個 stage 的輸出都可以 dump 成文字檔，之後 mismatch 時可以逐層回查。

4. Debug Flow

Testbench 會輸出三個中間 dump：

feature_mem0_dump.txt
feature_mem1_dump.txt
feature_mem2_dump.txt

若 final output mismatch，TB 會印出：

pixel index
RTL output
golden output
Conv1 output
Conv3 output
PixelShuffle output

這樣可以很快判斷問題是在 Conv1、Conv3、PixelShuffle，還是 final output formatting。 目前這比直接看整條 pipeline waveform 更適合學習和 debug。

5. End-to-End Verification

TB 讀取：

input_int8_for_core.mem
conv1_weight.mem
conv1_m0.mem
conv1_m1.mem
conv3_weight.mem
conv3_m0.mem
conv3_m1.mem
output_uint8.mem

完整驗證流程：

input_int8_for_core.mem
  -> sr_core_top
  -> output_mem
  -> compare output_uint8.mem
mismatch count = 0
max abs diff   = 0
PASS

6. 為什麼目前不引入 BRAM IP

BRAM IP 會帶入額外議題，例如 read latency、write-first/read-first mode、 byte enable、dual-port collision、address timing。這些都很重要， 但不應該和目前的 compute correctness 混在一起。

目前用 behavioral stage memory，是為了先確認 tensor value 與 tensor layout 完全正確。 等 Phase7 穩定後，下一步才適合把 feature memory 換成更接近 FPGA deployment 的 memory subsystem。

7. 後續演進

之後可以依序把 verification top 演進成 deployment top：

behavioral feature memory
  -> explicit SRAM/BRAM style memory
  -> Vivado BRAM IP
  -> weight BRAM / parameter BRAM
  -> AXI-Lite control
  -> AXI DMA input/output transfer

目前 Phase7 的價值是建立一個可信的 end-to-end RTL baseline。 之後任何 memory 或 bus integration 修改，都可以回來跟這個 baseline 比對。
