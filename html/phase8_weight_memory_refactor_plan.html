Phase8 Weight Memory Architecture Refactor

文件位置：model_lite/sr_core/html/phase8_weight_memory_refactor_plan.html

RTL_sys 草稿：model_lite/sr_core/RTL_sys/phase8_weight_memory/

返回：RTL_sys_index.html

Phase8 是 sr_core_top 完成 end-to-end verification 之後的第一個 accelerator system architecture refactor。這一階段先做 architecture review 與 weight memory proposal， 不修改已通過驗證的 RTL/。

1. Phase 目標

本階段要解決的是：目前 sr_core_top 為了 verification 方便， 把 Conv1 weight、Conv3 weight、M0、M1 全部攤平成 top-level input ports。 這種方式已經證明功能正確，但不是合理的 accelerator architecture。 

Phase8 goal:
large top-level parameter ports
  -> memory-style weight / quant parameter organization

Phase8 目前不做 AXI、DMA、BRAM IP、AXI BRAM Controller、PS-PL integration。 本階段只聚焦於 Weight Memory Architecture Refactor。

2. 現有問題

目前 verified top 內存在大量 weight/quant parameter input ports，例如：

conv1_w000
conv1_w001
...
conv1_w708

conv3_w00
...
conv3_w37

conv1_m0_0 ... conv1_m0_7
conv1_m1_0 ... conv1_m1_7

conv3_m0_0 ... conv3_m0_3
conv3_m1_0 ... conv3_m1_3
項目 目前方式 問題
Conv1 Weight Interface 72 個 signed int8 input ports Top-level port 過多，不利於封裝 IP，也不利於 BRAM/ROM 化。
Conv3 Weight Interface 32 個 signed int8 input ports Conv3 是 1x1 kernel，但目前仍然被攤平成大量 port。
M0/M1 Interface 每個 output channel 一組 port 未來若增加 layer 或 channel，port 數會快速膨脹。
Top-level Interface TB 直接把所有參數接進 DUT 適合 verification，不適合 accelerator deployment。

3. 為什麼不能維持舊架構

舊架構最大的優點是直覺：每個 weight 都清楚可見，適合早期 debug。 但當設計開始往 accelerator architecture 前進時，它會變成限制。 

舊架構限制：
1. Top-level port 太多
2. 不容易包成 Vivado custom IP
3. 不容易替換成 BRAM / ROM
4. 不容易建立 AXI weight loading path
5. 不容易支援未來更大模型或更多 channel
6. 不容易把 weight storage 和 compute datapath 分開

因此 Phase8 不應該繼續增加 port，而應該開始建立 memory-oriented parameter interface。 這是後面 BRAM Integration、AXI-Lite Integration、DMA Integration 的前置工作。

4. 新架構設計理念

Phase8 選擇 weight memory organization，而不是繼續維持 port-per-weight。 核心想法是：weight 和 quant parameter 是 model constant，應該像 memory 一樣被讀取， 不應該全部變成 top-level wires。

替代方案比較

方案 優點 缺點 Phase8 是否採用
維持大量 input port 最容易接現有 TB 不適合 IP / BRAM / AXI / deployment 否
直接上 Vivado BRAM IP 接近 deployment 太早引入 IP latency、XCI、初始化格式問題 否
先做 Verilog memory bank 保留可讀性，容易和 .mem golden 比對，未來可替換成 BRAM 還不是最終 deployment version 是

Phase8 採用 memory bank proposal：先用 Verilog memory interface 表達 weight organization， 不直接上 BRAM IP。這樣可以保持目前 PASS baseline，同時為未來 BRAM/AXI 遷移鋪路。

6. Weight Memory Concept

目標是把 weight 從 port list 改成 memory organization：

舊：
conv1_w000, conv1_w001, ..., conv1_w708

新：
reg signed [7:0] conv1_weight_mem [0:71]
舊：
conv3_w00, conv3_w01, ..., conv3_w37

新：
reg signed [7:0] conv3_weight_mem [0:31]

Compute datapath 未來可以透過 address 讀 weight。 若之後要轉成 BRAM，這種 address/data interface 也比較接近實際硬體 memory。

7. Conv1 Memory Mapping

Conv1 weight shape：

[8][3][3][1]

因為 Cin=1，所以可視為每個 output channel 有 9 個 kernel tap：

co0: addr  0.. 8
co1: addr  9..17
co2: addr 18..26
co3: addr 27..35
co4: addr 36..44
co5: addr 45..53
co6: addr 54..62
co7: addr 63..71

地址公式：

addr = output_channel * 9 + kernel_tap

其中 kernel_tap 對應 3x3 window 的 0..8， 順序需保持和 generated/conv1_weight.mem 一致。

8. Conv3 Memory Mapping

Conv3 是 layer 名稱，kernel 是 1x1。Conv3 weight shape：

[4][1][1][8]

每個 output channel 有 8 個 input channel weight：

co0: addr  0.. 7
co1: addr  8..15
co2: addr 16..23
co3: addr 24..31

地址公式：

addr = output_channel * 8 + input_channel

這個 mapping 和目前 generated/conv3_weight.mem 的 cout-major flatten layout 對齊。

9. M0/M1 Memory Mapping

M0/M1 是 per-output-channel quantization parameter。

Layer M0 count M1 count Mapping
Conv1 8 8 conv1_m0_mem[co], conv1_m1_mem[co]
Conv3 4 4 conv3_m0_mem[co], conv3_m1_mem[co]

未來可選擇兩種 memory organization：

方案 A：M0 和 M1 分開 memory
conv1_m0_mem[0:7]
conv1_m1_mem[0:7]

方案 B：每 channel 打包成 parameter record
channel -> M0, M1

Phase8 草稿先採用方案 A，因為它比較直覺，也比較容易和目前 conv*_m0.mem、 conv*_m1.mem 對照。

10. 驗證方式

Phase8 目前是 architecture proposal，因此第一步驗證不是 end-to-end RTL 替換，而是確認 memory mapping 正確。

未來正式驗證會使用：

generated/conv1_weight.mem
generated/conv3_weight.mem
generated/conv1_m0.mem
generated/conv1_m1.mem
generated/conv3_m0.mem
generated/conv3_m1.mem
generated/golden/conv1_out_int8.mem
generated/golden/conv3_out_int8.mem
generated/golden/output_uint8.mem

建議分兩層驗證：

驗證層級 目的 Golden

System wrapper test 確認新的 memory-oriented top 仍然輸出正確結果 output_uint8.mem

11. 驗證流程

未來 Phase8 implementation 的建議 data flow：

TB
  -> load conv1_weight.mem into conv1_weight_mem
  -> load conv3_weight.mem into conv3_weight_mem
  -> load conv1_m0/m1 and conv3_m0/m1 into quant memories
  -> load input_int8_for_core.mem
  -> run RTL_sys DUT
  -> compare final output with output_uint8.mem
  -> optionally dump intermediate feature memories

這個 flow 要保留目前 PASS 版本的驗證精神： 每次 architecture refactor 後，都要能回到 golden compare。

12. PASS 條件

Phase8 最終實作完成後，PASS 條件仍然是：

mismatch count = 0
max abs diff   = 0
PASS

另外 memory mapping 層級也應該檢查：

all conv1_weight_mem values match conv1_weight.mem
all conv3_weight_mem values match conv3_weight.mem
all M0/M1 memories match generated parameter files

13. 遇到的問題

目前 architecture review 已確認的問題：

1. Top-level port 過多
2. Weight interface 不合理
3. M0/M1 interface 不利於擴充
4. Verification top 和 deployment architecture 邊界混在一起
5. 若直接導入 BRAM/AXI，debug 複雜度會太早上升

因此 Phase8 採用中間路線：先做 memory-style proposal， 讓 weight interface 逐步接近硬體 memory，但不立刻引入 BRAM IP。

14. 未來 BRAM Migration Plan

Phase8 的 memory bank proposal 未來可以逐步遷移成 BRAM：

Verilog reg memory proposal
  -> explicit sync ROM/RAM interface
  -> Vivado Block Memory Generator
  -> COE/MEM initialization
  -> synthesis/deployment integration

BRAM 化時需要注意：

1. read latency
2. address timing
3. signed interpretation
4. initialization format
5. one-port or dual-port requirement
6. Conv1/Conv3 是否需要 parallel weight reads

15. 未來 AXI Migration Plan

AXI weight loading 不在 Phase8 實作，但 Phase8 的 memory organization 會讓未來 AXI loading 比較合理。 

AXI-Lite option:
  PS writes weight/parameter register or small memory window

AXI DMA option:
  PS transfers weight blob to BRAM before inference

Fixed ROM option:
  weights are fixed at bitstream generation time

對目前 QAT fixed model 而言，最簡單 deployment path 可能是 fixed ROM。 若未來要支援 runtime weight update，再考慮 AXI loading。

16. 最終結論

本階段完成：

1. 確認 RTL/ 保留為 Golden RTL Version
2. 建立 RTL_sys/phase8_weight_memory/
3. 建立 weight memory architecture proposal
4. 定義 Conv1 / Conv3 / M0 / M1 memory mapping
5. 定義未來 verification strategy
6. 建立未來 BRAM / AXI migration 方向

本階段尚未完成：

1. 尚未把 sr_core_top 改成 memory-oriented top
2. 尚未建立完整 Phase8 self-checking TB
3. 尚未替換任何 verified RTL module
4. 尚未導入 BRAM IP / AXI / DMA

Phase8 下一步可以開始建立 memory-oriented wrapper 或 top， 驗證 memory bank 讀出的 weight/M0/M1 能否維持目前 end-to-end PASS。

Phase9 預計進入：

Streaming Architecture Refactor

Phase9 會處理 valid-ready、stage control、input/output stream、以及 feature data movement。 但在 Phase9 前，Phase8 必須先把 parameter interface 整理乾淨。

18. Phase8.1 Weight Memory Self-Check

Phase8 原本的 sr_weight_memory_arch_proposal.v 只是 architecture draft。 它定義了 memory-style organization，但 proposal 本身不代表 mapping 已經被驗證。 因此 Phase8.1 先建立獨立 self-check，不急著接 sr_core_top。

18.1 為什麼 proposal 還不等於完成

Memory architecture 的第一個風險不是 MAC 算錯，而是 address mapping 錯。 例如 Conv1 的 72 個 weight 若 output channel 或 kernel tap 順序錯掉， 後面即使 MAC 和 requant 都正確，end-to-end output 仍然會 mismatch。

所以 Phase8.1 只驗 memory readback，不做 Conv1、Conv3、PixelShuffle、OutputStage。 這樣可以把問題限制在 weight/M0/M1 memory mapping 本身。

18.2 使用檔案

model_lite/sr_core/generated/conv3_weight.mem
model_lite/sr_core/generated/conv1_m0.mem
model_lite/sr_core/generated/conv1_m1.mem
model_lite/sr_core/generated/conv3_m0.mem
model_lite/sr_core/generated/conv3_m1.mem

這些檔案都是 signed decimal text，因此 testbench 使用 $fopen 與 $fscanf 載入，不使用 $readmemh。

18.3 驗證方式

TB 先讀 generated 檔案作為 golden，然後用 hierarchical assignment 把 golden 寫入 proposal DUT 內部 memory。接著逐 address 設定讀取位址， 等同步 readback 後比較 DUT output 和 golden value。 

TB
  -> read generated/*.mem
  -> load DUT memory by hierarchical assignment
  -> drive address
  -> read DUT output
  -> compare with golden value

18.7 目前尚未接 sr_core_top

Phase8.1 只確認 memory-style organization 的 mapping 正確。 它尚未把 memory proposal 接到 sr_core_top， 也尚未建立 memory-oriented top wrapper。 這是刻意保留的邊界，避免 mapping 問題和 system integration 問題混在一起。

18.8 下一步 Phase8.2

Phase8.2 的目標是建立 Memory-Oriented Top Wrapper。 也就是在不修改 RTL/ golden verified modules 的前提下， 建立 RTL_sys 版本的 wrapper，讓 weight/M0/M1 從 memory-style interface 餵進 compute blocks， 並重新確認 end-to-end output 仍然 PASS。

19. Phase8.2 Memory-Oriented Top Wrapper

Phase8.2 在 Phase8.1 mapping self-check 通過後，建立 sr_core_top_mem_wrapper.v。這個 wrapper 的目標是把外部大量 weight/M0/M1 ports 收進內部 memory-style arrays，同時仍然使用已 verified 的 sr_core_top 與下層 compute modules。

19.1 為什麼需要 wrapper

原本的 sr_core_top 是 verification-oriented top。 它把每個 weight 和 quant parameter 都攤成 input port，方便早期 TB 直接接線。 但 accelerator architecture 需要的是 memory boundary，而不是大量固定 port。 

舊方式：
TB -> many weight/M0/M1 ports -> sr_core_top

Phase8.2：
TB -> wrapper memories -> verified sr_core_top -> output_uint8

19.2 為什麼不直接修改 verified RTL

model_lite/sr_core/RTL 目前是 Golden RTL Version。 直接修改 sr_core_top 會讓已通過的 Phase7 baseline 失去穩定參考。 因此 Phase8.2 選擇在 RTL_sys 內建立 wrapper，而不是改動 verified RTL。

這個做法保留兩個版本：RTL/sr_core_top 是已驗證 baseline； RTL_sys/phase8_weight_memory/sr_core_top_mem_wrapper.v 是 memory-oriented architecture refactor 版本。

19.3 Wrapper 和 sr_core_top 的差異

項目 sr_core_top sr_core_top_mem_wrapper
用途 Verification-oriented top Memory-oriented architecture wrapper
Weight/M0/M1 大量 top-level input ports 內部 memory-style arrays
Compute datapath 直接使用 verified blocks 仍然使用 verified sr_core_top，不重寫 datapath
部署接近度 適合 simulation verification 比較接近未來 ROM/BRAM/AXI parameter storage

19.7 遇到的問題與設計取捨

最大問題是：若現在直接重寫 sr_core_top 成真正 memory-read architecture， 會同時改動 control timing、parameter access timing、stage flow，debug 風險太高。 因此 Phase8.2 先用 wrapper 方式，把外部 interface 改成 memory-oriented， 但內部 compute 行為仍由 verified top 保證。

方案 結果 原因
直接修改 verified sr_core_top 不採用 會破壞 Golden RTL baseline。
直接導入 BRAM IP 不採用 太早加入 read latency、XCI、init file 等部署問題。
建立 memory-oriented wrapper 採用 能降低 top-level ports，又能維持目前 PASS 行為。
