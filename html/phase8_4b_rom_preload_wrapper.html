文件位置：model_lite/sr_core/html/phase8_4b_rom_preload_wrapper.html

本階段把 Phase8.3 已建立的 Vivado BMG ROM 參數來源，正式接到 memory-oriented top wrapper。 目標不是只證明 ROM 會吐資料，而是驗證： 

Vivado ROM IP
-> preload FSM
-> parameter register bank
-> verified RTL pipeline
-> output_uint8.mem

Phase8.4b 結果：PASS。 mismatch count = 0，max abs diff = 0。

1. Phase8.4b 目標

Phase8.2 使用 behavioral register memory 直接保存 weight、M0、M1。 Phase8.4b 將來源替換成 ROM IP 讀出，再 preload 到 register bank，最後餵給已驗證的 sr_core_top。

2. 為什麼不每次都從 ROM 取參數？

Conv1 MAC 和 Conv3 MAC 目前是清楚、平行、教學型 datapath。 如果每個 MAC 運算點都即時從 ROM 取 weight，會立刻遇到兩個問題：

  • Vivado BMG ROM 是 synchronous read，address 送出後下一拍 data 才有效。
  • Conv1 一次需要 72 個 weight、Conv3 一次需要 32 個 weight，直接即時讀 ROM 會讓 MAC timing 和 control 複雜化。

因此本階段採用 preload：啟動 core 前，先把 ROM 內容完整搬到 parameter register bank。 這樣後面的 Conv1 / Conv3 / requant 還能維持 Phase7 已驗證的接口與行為。

3. ROM Latency 問題

Phase8.4a 已確認 Vivado BMG ROM 的 read latency 是 1 clock。 也就是： 

cycle N:
  rom_en   = 1
  rom_addr = A

cycle N+1:
  rom_dout = mem[A]

所以 preload FSM 不能用 same-cycle data 思維。 正確流程要拆成送 address、等待 ROM、擷取 data。

4. Preload FSM 設計

本階段的 FSM 使用明確狀態，不用硬編碼 delay，也不假設 data 同拍有效。 

IDLE
  -> C1W_SEND
  -> C1W_WAIT
  -> C1W_CAPTURE
  -> C3W_SEND
  -> C3W_WAIT
  -> C3W_CAPTURE
  -> C1M_SEND
  -> C1M_WAIT
  -> C1M_CAPTURE
  -> C3M_SEND
  -> C3M_WAIT
  -> C3M_CAPTURE
  -> DONE

preload_done assert 後，wrapper 才允許 core_start 進入 verified sr_core_top。這避免 core 在參數尚未載入完成時開始計算。

這個 timing 是 Phase8.4b 的核心。 只要 address 與 data 沒對齊，weight 或 M0/M1 就會晚一拍寫錯位置，最後 output_uint8 會 mismatch。

10. 遇到的問題

  • 前一次模擬失敗不是 datapath mismatch，而是 testbench 與 ROM init file 的相對路徑沿用自 Phase8.3。
  • Phase8.4b 的模擬工作目錄不同，所以需要把 input/golden/result/dump 路徑改成相對於本資料夾。
  • 最後改用 Vivado project 產生的 BMG ROM simulation wrapper 重新驗證，並把對應 .mif 初始化檔放在 xsim 工作目錄。

Vivado 的 BMG simulation model 本身仍會顯示 behavioral model 訊息，這是 Xilinx IP 的模擬模型，不是重新手寫的本地 ROM。 本次最終 PASS 是用 sr_core_bram_pynqz2.gen/sources_1/ip/*/sim/*.v 這些既有 IP wrapper 跑出的結果。

11. What did we learn?

Phase8 的演進重點是逐步把「驗證容易」的架構，改成「未來可部署」的架構： 

Behavioral Memory
-> Vivado ROM IP
-> Parameter Register Bank
-> Verified RTL

最大學習是：ROM IP 不是一般組合邏輯查表。 只要換成 synchronous ROM，就必須明確管理 address、enable、data valid。 先 preload 到 register bank 是目前最穩的方案，因為它保留了所有已 PASS 的 RTL 行為，也為後續 BRAM / AXI / DMA 留出清楚邊界。
