Phase8.1 Weight Memory Self-Check

文件位置：model_lite/sr_core/html/phase8_1_memory_mapping_selfcheck.html

返回：Phase8 完整紀錄

Phase8.1 的任務是確認 memory-style organization 的 address mapping 是否正確。 這一步只驗 weight/M0/M1 memory readback，尚未接 sr_core_top。

1. 為什麼 Proposal 還不等於完成

sr_weight_memory_arch_proposal.v 只是 architecture draft。 它定義了 Conv1 weight、Conv3 weight、M0/M1 的 memory-style organization， 但還沒有證明 address mapping 和 generated/*.mem 完全一致。

如果 memory address 順序錯，即使 MAC、requant、PixelShuffle 都是正確的， 最終 output 仍然會 mismatch。因此 Phase8.1 先只驗 memory mapping，不混入其他 stage。

2. 使用檔案

model_lite/sr_core/generated/conv1_weight.mem
model_lite/sr_core/generated/conv3_weight.mem
model_lite/sr_core/generated/conv1_m0.mem
model_lite/sr_core/generated/conv1_m1.mem
model_lite/sr_core/generated/conv3_m0.mem
model_lite/sr_core/generated/conv3_m1.mem

這些檔案都是 signed decimal text，因此 testbench 使用 $fopen 與 $fscanf 載入，不使用 $readmemh。

3. 驗證方式

TB
  -> read generated/*.mem as golden
  -> load DUT internal memories by hierarchical assignment
  -> drive address sequentially
  -> read memory module output
  -> compare output data with golden value

目前載入方式刻意使用 hierarchical assignment，例如 dut_conv1_weight.conv1_weight_mem[i] = golden_value。 這是為了先把驗證焦點鎖定在 mapping，不提前導入 BRAM IP 或 loader FSM。

4. Mapping 規則

Memory Address Rule Count
Conv1 weight addr = output_channel * 9 + kernel_tap 72
Conv3 weight addr = output_channel * 8 + input_channel 32
Conv1 M0/M1 addr = output_channel 8 each
Conv3 M0/M1 addr = output_channel 4 each

5. PASS 條件與結果

conv1_weight mismatch count = 0
conv3_weight mismatch count = 0
conv1_m0 mismatch count     = 0
conv1_m1 mismatch count     = 0
conv3_m0 mismatch count     = 0
conv3_m1 mismatch count     = 0
total mismatch count        = 0
PASS
