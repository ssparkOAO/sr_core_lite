Phase9.3 Reference-Like CNN Hierarchy

文件位置：model_lite/sr_core/html/phase9_3_ref_hierarchy.html

工作區：model_lite/sr_core/RTL_sys/phase9_3_ref_hier/

Phase9.3 的目標是整理 CNN hierarchy，讓 Vivado schematic 看起來更接近 ref/cnn_classfication：controller、Conv1、Conv3、ROM/RAM block 在同一層可以直接看見。這一階段使用 ram_ip_bhv.v 行為模型驗證， 還不是正式的 Vivado BMG IP-backed simulation。

1. 為什麼做 Phase9.3

之前的 wrapper 功能正確，但層級太像「把所有東西包起來的驗證外殼」。 你希望最後在 Vivado schematic 裡看到的結構，是 CNN block、controller、 ROM/RAM IP 同層排列，像 reference project 那樣容易追線、容易定位 module。

因此 Phase9.3 不改 verified RTL 本體，而是重組 system-level top： Conv1 和 Conv3 不合併，兩者仍是獨立 CNN block，中間用一個小的 register slice 做 valid/data 對齊，並且移除 feature_mem0。

2. Target Hierarchy

sr_nn_top
├─ sr_ctrl
├─ sr_pctrl
├─ conv1_weight_rom
├─ conv3_weight_rom
├─ conv1_m0_rom
├─ conv1_m1_rom
├─ conv3_m0_rom
├─ conv3_m1_rom
├─ conv1_block
├─ sr_c1c3_slice
├─ conv3_block
├─ conv3_feature_ram
├─ pixel_shuffle_core
├─ sr_out_pack
└─ output_image_ram

這個 hierarchy 的重點不是最高效能，而是讓 PL 端 CNN datapath 和 memory block 的關係清楚可見。這會讓後續換成 Vivado IP、看 schematic、做 synthesis check 都比較容易。

3. 資料夾整理

資料夾 用途
rtl/ Phase9.3 reference-like hierarchy RTL：sr_nn_top、sr_ctrl、sr_pctrl、sr_c1c3_slice、sr_out_pack。
sim/ Behavioral-only simulation：tb_sr_nn_top.v 與 ram_ip_bhv.v。這裡不是 Vivado BMG IP。
results/ Phase9.3 behavioral PASS 結果與 dump 檔。
work/ xsim 產生的 log、snapshot、.wdb 波形資料。這些不是 RTL source。

4. Phase9.3 驗證方式

Phase9.3 使用手寫的 ram_ip_bhv.v 模擬 ROM/RAM module name。 它的用途是先確認新 hierarchy、controller、Conv1→Conv3 streaming、 PixelShuffle write transaction、output packing 這些邏輯正確。 

TB input source
→ sr_nn_top
→ behavioral ROM/RAM model
→ Conv1
→ Conv1-to-Conv3 register slice
→ Conv3
→ behavioral conv3_feature_ram
→ PixelShuffle
→ sr_out_pack
→ behavioral output_image_ram
→ compare golden

PASS 條件是 conv3_feature_ram 對上 conv3_out_int8.mem， 且 output_image_ram 對上 output_uint8.mem， mismatch count 與 max abs diff 都為 0。

5. Phase9.3 和 Phase9.4 的分界

Phase Memory 目的
Phase9.3 ram_ip_bhv.v 先確認新 hierarchy 功能正確。
Phase9.4 Vivado BMG IP wrapper 確認同一個 sr_nn_top 可以接真正 Vivado ROM/RAM IP simulation wrapper。

6. What Did We Learn?

CNN block 不一定要包進一個巨大 wrapper 才能工作。更適合 deployment 前期的做法， 是讓 controller、CNN block、ROM/RAM IP 以清楚層級並列，保留 verified module 的可追蹤性，也讓 Vivado schematic 更接近真實 PL architecture。
