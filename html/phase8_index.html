Phase8 Weight Memory Architecture Refactor

文件位置：model_lite/sr_core/html/phase8_index.html

RTL_sys：model_lite/sr_core/RTL_sys/phase8_weight_memory/

返回：RTL_sys_index.html

Phase8 是 sr_core_top 完成 end-to-end verification 後的第一個 accelerator system architecture refactor。它的核心目標是把大量 weight/M0/M1 top-level ports 逐步改成 memory-style architecture，同時保留 RTL/ 內已通過驗證的 Golden RTL Version。

1. Phase 目標

原本 sr_core_top 為了 verification 方便，把 Conv1 weight、Conv3 weight、 M0、M1 都攤平成 top-level input ports。這已經證明功能正確，但不適合 accelerator architecture。 

Phase8 goal:
large top-level parameter ports
  -> memory-style weight / quant parameter organization

本階段不做 AXI、DMA、BRAM IP、AXI BRAM Controller、PS-PL integration。 目前只處理 Weight Memory Architecture Refactor。

M0/M1 Interface 每個 output channel 一組 port 未來若增加 layer 或 channel，port 數會快速膨脹。

3. 為什麼不能維持舊架構

1. Top-level port 太多
2. 不容易包成 Vivado custom IP
3. 不容易替換成 BRAM / ROM
4. 不容易建立 AXI weight loading path
5. 不容易支援未來更大模型或更多 channel
6. 不容易把 weight storage 和 compute datapath 分開

舊架構適合 verification，但不適合 deployment-oriented accelerator。 Phase8 必須開始建立 memory boundary，否則後面的 BRAM/AXI/DMA 會很難接。

4. 新架構設計理念

Weight 和 quant parameter 是 model constant，應該像 memory 一樣被讀取， 不應該全部變成 top-level wires。

方案 優點 缺點 採用
維持大量 input port 最容易接現有 TB 不適合 IP / BRAM / AXI / deployment 否
直接上 Vivado BRAM IP 接近 deployment 太早引入 latency、XCI、初始化格式問題 否
先做 Verilog memory bank / wrapper 可讀、可驗證、未來可替換 BRAM 還不是 deployment version 是

6. Weight Memory Concept

舊：
conv1_w000, conv1_w001, ..., conv1_w708

新：
reg signed [7:0] conv1_weight_mem [0:71]
舊：
conv3_w00, conv3_w01, ..., conv3_w37

新：
reg signed [7:0] conv3_weight_mem [0:31]

這種 memory-style organization 讓後續 BRAM/ROM/AXI migration 有明確的 address/data boundary。

7. Conv1 Memory Mapping

Conv1 weight shape 為 [8][3][3][1]，因為 Cin=1，所以每個 output channel 有 9 個 kernel tap。

co0: addr  0.. 8
co1: addr  9..17
co2: addr 18..26
co3: addr 27..35
co4: addr 36..44
co5: addr 45..53
co6: addr 54..62
co7: addr 63..71
addr = output_channel * 9 + kernel_tap

8. Conv3 Memory Mapping

Conv3 是 layer 名稱，kernel 是 1x1。Conv3 weight shape 為 [4][1][1][8]。

co0: addr  0.. 7
co1: addr  8..15
co2: addr 16..23
co3: addr 24..31
addr = output_channel * 8 + input_channel

9. M0/M1 Memory Mapping

Layer M0 count M1 count Mapping
Conv1 8 8 conv1_m0_mem[co], conv1_m1_mem[co]
Conv3 4 4 conv3_m0_mem[co], conv3_m1_mem[co]

10. 驗證策略

Phase8 的重點不是重新證明 MAC、requant、PixelShuffle 這些運算正確，因為那些已經在 RTL/ 內完成 golden verification。Phase8 要驗證的是：把大量 top-level weight/M0/M1 port 改成 memory-style organization 之後，資料 mapping 沒有錯，而且整個 end-to-end 行為仍然和原本 PASS 版本一致。

因此 Phase8 分成兩層驗證。第一層先只看 memory 本身，第二層才把 memory 接回 top wrapper 做完整輸出比對。這樣如果未來出錯，可以很快知道問題是在「memory mapping」還是「wrapper 接線 / 系統整合」。

階段 目的 重點說明 使用檔案 / PASS 意義
Phase8.1 Weight Memory Self-Check 只驗證 memory-style organization 的 mapping 與 readback，不接 sr_core_top。這一層要確認 conv1_weight_mem[0:71]、conv3_weight_mem[0:31]、conv1_m0/m1、conv3_m0/m1 的 address 對應完全正確。這一步很重要，因為只要 weight channel 順序、kernel tap 順序、M0/M1 output channel 對應錯一格，後面 end-to-end 可能只看到 output mismatch，很難立即定位。 generated/conv1_weight.mem, generated/conv3_weight.mem, generated/conv1_m0.mem, generated/conv1_m1.mem, generated/conv3_m0.mem, generated/conv3_m1.mem
PASS 代表：generated 參數載入 memory 後，逐地址讀回完全一致，memory mapping 本身可信。
Phase8.2 Memory-Oriented Top Wrapper 在不修改 RTL/ verified modules 的前提下，建立 sr_core_top_mem_wrapper。Wrapper 內部持有 weight/M0/M1 memory，再把 memory entry 展開接到原本 verified top 的 ports。這一層驗證的是：新的 memory-oriented architecture 是否能保持原本 end-to-end 行為，也就是 refactor 沒有破壞 golden result。 generated/golden/input_int8_for_core.mem, generated/golden/output_uint8.mem，並保留 feature_mem0_dump.txt, feature_mem1_dump.txt, feature_mem2_dump.txt
PASS 代表：從 input_int8 經 Conv1、Conv3、PixelShuffle、OutputStage 到 output_uint8，最終結果仍然 bit-exact match golden。

這個兩層驗證策略的好處是保留 debug 階層：Phase8.1 先保證參數 memory 沒有放錯；Phase8.2 再保證 wrapper 接回 verified pipeline 後輸出不變。未來若要換成 BRAM IP、AXI loader 或 DMA loading，也應該維持同樣節奏：先 readback 自檢，再 end-to-end golden compare。

13. 遇到的問題

1. Top-level port 過多
2. Weight interface 不合理
3. M0/M1 interface 不利於擴充
4. Verification top 和 deployment architecture 邊界混在一起
5. 若直接導入 BRAM/AXI，debug 複雜度會太早上升

14. BRAM Migration Plan

Verilog reg memory proposal
  -> explicit sync ROM/RAM interface
  -> Vivado Block Memory Generator
  -> COE/MEM initialization
  -> synthesis/deployment integration

15. AXI Migration Plan

AXI-Lite option:
  PS writes weight/parameter register or small memory window

AXI DMA option:
  PS transfers weight blob to BRAM before inference

Fixed ROM option:
  weights are fixed at bitstream generation time

對目前 fixed QAT model 而言，最簡單的 deployment path 可能是 fixed ROM。 如果未來要支援 runtime weight update，再考慮 AXI loading。

21. Phase8.4a ROM Timing Analysis

本章保留 Phase8.4a 在 Phase8 完整紀錄中的位置。 詳細說明已獨立整理到： phase8_4a_rom_timing_analysis.html

Phase8.4a 的重點是確認：BMG ROM 不是 same-cycle behavioral memory。 Address / enable 在 cycle N 送出，data 會在 cycle N+1 有效。 因此 Phase8.4b 的 ROM-backed wrapper 必須先處理 preload 或 wait timing， 不能直接把 ROM data 當成同拍可用。
