Phase6 Output Stage

回到主頁：SR Core 專案首頁

回到 RTL 進度：rtl/index.html

本頁記錄 final output formatting stage： signed int8 → output clip → uint8 image domain。

1. 為什麼 PixelShuffle Output 還是 Signed Int8

前面 Conv1、Conv3、PixelShuffle 都還在 quantized tensor domain。 PixelShuffle 只重新排列位置，不改變數值格式，因此 pixel_shuffle_out_int8.mem 仍然是 signed int8。

2. 為什麼需要 Output Clip

TFLite graph 在 final output quantize 前有 output range constraint。 硬體上先把數值限制在 signed int8 有效範圍： 

[-128, 127]

目前 input 已經是 signed int8，所以 clip 通常是 passthrough； 但保留明確 clamp datapath 可以讓 final stage 的 range constraint 很清楚。

3. 為什麼需要 Int8 到 Uint8 Conversion

SR core 內部使用 signed int8 quantized tensor。 但最終影像輸出 output_uint8.mem 是 uint8 image domain， 範圍是： 

[0, 255]

因此最後一步要做 zero-point recovery。

4. Zero Point = -128 的意義

目前 signed int8 tensor 的 zero point 是 -128。 要轉回 uint8 image domain，就把 signed int8 值平移 128： 

output_uint8 = clipped_int8 + 128
-128 -> 0
0    -> 128
127  -> 255

5. Pipeline 位置

Conv1 block
  -> Conv3 block
  -> PixelShuffle
  -> Output Clip
  -> int8 to uint8 conversion
  -> output_uint8

這不是一般 CNN layer，而是 final output formatting stage。

6. Golden Verification

Testbench 直接使用完整 golden：

generated/golden/pixel_shuffle_out_int8.mem
  -> sr_output_stage
  -> compare generated/golden/output_uint8.mem

若 mismatch，會印出：

pixel index
input int8
clipped value
RTL uint8 output
golden uint8 output

7. PASS 條件

mismatch count = 0
max abs diff   = 0

8. 目前狀態

完成 Phase6 後，Conv1 → Conv3 → PixelShuffle → Output Stage 已經接近完整 SR accelerator 的 pure RTL compute path。
