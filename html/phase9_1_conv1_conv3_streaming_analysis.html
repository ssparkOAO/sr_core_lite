Phase9.1 Conv1 → Conv3 Streaming Analysis

文件位置：model_lite/sr_core/html/phase9_1_conv1_conv3_streaming_analysis.html

參考架構：model_lite/sr_core/RTL_sys/phase8_4_rom_preload_wrapper/sr_core_top_rom_preload_wrapper.v

參考 verified top：model_lite/sr_core/RTL/sr_core_top/sr_core_top.v

Phase9 開始 Streaming Architecture Refactor。Phase9.1 只分析如何消除 feature_mem0，不寫 RTL、不修改 RTL/ 已驗證模組。

結論：feature_mem0 可以移除。原因是 Conv3 是 1x1 convolution， 對每個 pixel 只需要同一個 pixel 的 8-channel Conv1 output，不需要完整 feature map。

1. 現有架構

目前 Phase8.4b 的外層 wrapper 先完成 ROM preload，將 weight / M0 / M1 載入 parameter register bank， 再啟動 verified sr_core_top。而 sr_core_top 內部仍是 verification-oriented stage memory bridge 架構。 

Input
-> Conv1 block
-> feature_mem0
-> Conv3 block
-> feature_mem1
-> PixelShuffle
-> feature_mem2
-> OutputStage
-> output_uint8

這種方式對 Phase7 / Phase8 很有價值，因為每個 stage 都可以 dump 並與 golden 對齊。 但它不是最後 accelerator architecture，因為每一層都要等前一層整張 feature map 完成。

2. feature_mem0 作用

在目前 sr_core_top.v 中，feature_mem0 是 Conv1 和 Conv3 之間的中間 feature map。 

reg signed [7:0] feature_mem0 [0:511];

它保存：

項目內容
Shape8 x 8 x 8
元素數64 pixels * 8 channels = 512 bytes
寫入來源conv1_q0 ~ conv1_q7
寫入時機conv1_out_valid == 1
地址(conv1_out_y * 8 + conv1_out_x) * 8 + channel
讀出用途Conv3 在 ST_CONV3 逐 pixel 讀 8 channels

現有控制流程是先跑完整張 Conv1，再切到 Conv3： 

ST_CONV1:
  if conv1_out_valid:
    write feature_mem0
    if last pixel:
      state -> ST_CONV3

ST_CONV3:
  read feature_mem0[pixel*8 + 0..7]
  run Conv3
  write feature_mem1

3. 為什麼可移除

Conv3 是 1x1 convolution。它的數學是： 

acc[co] =
  sum ci=0..7 input[ci] * weight[co][ci]

對於某一個 pixel，Conv3 只需要這個 pixel 的 8 個 Conv1 output channel。 它不需要上、下、左、右 pixel，也不需要完整 8x8 feature map。

Layer是否需要鄰近 pixel原因
Conv1 3x3需要kernel 3x3，需要 window / line buffer / padding
Conv3 1x1不需要只混合同一 pixel 的 8 channels

因此 feature_mem0 目前保存的 512 bytes 中，Conv3 真正同一時間需要的只有 conv1_q0 ~ conv1_q7 這一組 8 bytes。

4. Streaming 資料流

Phase9.1 建議的方向是把 Conv1 output valid 直接接到 Conv3 input valid。 資料不再先進 feature_mem0，而是形成 pixel-level streaming。 

Input stream
-> Conv1 block
   out_valid, out_x, out_y, q0~q7
-> optional register slice
-> Conv3 block
   q0~q3
-> feature_mem1

注意：Phase9.1 只消除 feature_mem0。 feature_mem1 和 feature_mem2 可以先保留，讓後續 verification 仍然容易。

第一版 streaming refactor 不需要一次把 Conv3、PixelShuffle、OutputStage 全部串成 streaming。 正確策略是先消除一個 stage memory，驗證 PASS，再進下一段。

5. 控制訊號分析

Conv1 → Conv3 之間至少需要以下訊號：

Signal用途備註
conv1_out_valid表示 q0~q7 是有效 pixel可成為 Conv3 input valid
conv1_out_xConv1 output pixel x 座標用於 debug 或 feature_mem1 address
conv1_out_yConv1 output pixel y 座標用於 debug 或 feature_mem1 address
conv1_q0~q7Conv3 的 8-channel input同一 pixel 的 channel vector
conv3_out_validConv3 output q0~q3 有效如果 Conv3 是 combinational，可由 valid register 對齊
conv3_out_index寫 feature_mem1 的 pixel index可由 x/y 轉成 y*8+x

目前 sr_conv1x1_cin8_cout4_block 是 combinational MAC + requant block，沒有自己的 valid port。因此 Phase9.2 若實作 RTL_sys wrapper，需要在 wrapper 內建立 conv3_in_valid / conv3_out_valid 的對齊規則。

6. Valid Pipeline 分析

如果 Conv3 block 保持 combinational，最簡單的 valid 對齊方式如下： 

cycle N:
  conv1_out_valid = 1
  conv1_q0~q7     = valid
  conv3_q0~q3     = combinational result

cycle N+1:
  conv3_out_valid = registered conv1_out_valid
  write feature_mem1 with registered conv3_q0~q3

這等於在 Conv1 和 Conv3 之間加入一個 register slice。好處是：

  • Conv1 output 和 Conv3 input 邊界清楚。
  • Conv3 output valid 可以穩定對齊 feature_mem1 write enable。
  • 未來若 Conv3 加 pipeline，只要延長 valid delay chain。

如果不加 register slice，也可以同拍用 conv1_out_valid 寫入 Conv3 結果； 但這會讓 timing boundary 比較不清楚，也比較不利於之後擴展。

7. 是否需要 FIFO？

選項適用情境Phase9.1 建議
No FIFO Conv3 可以每個 Conv1 valid 都立即吃掉；下游先保留 feature_mem1。 建議第一版採用
Register Slice 需要切清楚 valid/data timing boundary。 建議加入
Small FIFO Conv1 和 Conv3 有 backpressure，或下游不能每 cycle 接收。 暫時不需要，可作為 Phase9 後續選項

目前 Conv3 是 1x1 combinational block，且 Phase9.1 後仍會把 Conv3 output 寫入 feature_mem1。因此暫時不需要 FIFO。 但建議使用 register slice，因為它是最低成本、最清楚的 timing 保護。

8. Verification Strategy

Phase9.1 本階段不寫 RTL，因此驗證策略是為 Phase9.2 先定義好。

驗證層級目標Golden
Conv1 output stream 確認 conv1_out_valid 對應的 q0~q7 與原本 feature_mem0 dump 一致 generated/golden/conv1_out_int8.mem
Conv1 → Conv3 stream 確認 Conv3 接到 Conv1 stream 後產生同樣 q0~q3 generated/golden/conv3_out_int8.mem
End-to-end 確認 final output 仍 bit-exact generated/golden/output_uint8.mem

建議 Phase9.2 testbench 保留 feature_mem1_dump.txt、 feature_mem2_dump.txt 與 final result txt。 但 feature_mem0_dump.txt 可以改成 optional stream dump，例如： 

conv1_stream_dump.txt:
pixel_index x y q0 q1 q2 q3 q4 q5 q6 q7

預期資料流：

這個 wrapper 仍可沿用 Phase8.4b 的 ROM preload / parameter register bank 概念， 但不應再例化原本有 feature_mem0 的 verified sr_core_top。 它應該在 RTL_sys 中重新組合已 verified 的 lower-level blocks。

11. 遇到的風險

風險說明建議處理
Valid/data misalignment Conv1 valid、q0~q7、x/y 若沒有一起 register，Conv3 寫入 index 可能錯位。 使用 register slice，同步保存 q0~q7、x、y、valid。
Conv3 combinational timing 變長 Conv1 output 後直接接 Conv3，可能讓一拍內的 combinational path 變長。 Conv3 output register 化，valid 延一拍。
Debug 少了 feature_mem0 移除 feature_mem0 後不能直接 dump 中間完整 memory。 新增 conv1_stream_dump.txt 作為替代。
過早全 streaming 一次移除 feature_mem0/1/2 會讓 mismatch 難定位。 Phase9.1/9.2 只移除 feature_mem0，其他 stage bridge 先保留。

12. What did we learn?

feature_mem0 原本不是錯誤設計，它是 verification-oriented top 的 debug bridge。 但當 Conv1 和 Conv3 都已獨立與 end-to-end PASS 後，這個 bridge 就變成 architecture overhead。

Streaming refactor 的第一個安全切入點是 Conv1 → Conv3，因為 Conv3 是 1x1 layer。 它沒有 spatial dependency，只需要同一 pixel 的 channel vector。

Phase9.1 的最重要結論： feature_mem0 可以由 conv1_out_valid + q0~q7 + x/y 的 pixel-level stream 取代。第一版不需要 FIFO，但建議加入 register slice。

下一步 Phase9.2 才開始寫 RTL_sys wrapper，並以 conv3_out_int8.mem 和 output_uint8.mem 做驗證。
