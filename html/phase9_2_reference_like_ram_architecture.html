Phase9.2 Reference-Like RAM/IP Architecture

文件位置：model_lite/sr_core/html/phase9_2_reference_like_ram_architecture.html

RTL 位置：model_lite/sr_core/RTL_sys/phase9_streaming_architecture/

Vivado 專案：model_lite/sr_core/vivado/sr_core_streaming_pynqz2/

Phase9.2 的重點是把 SR Core 從 verification-oriented stage memory bridge， 推進到接近 ref/cnn_classfication 的 PL 架構： CNN datapath 和 ROM/RAM IP 在同一層，由 controller 產生 raw memory port。

重要狀態：目前 Phase9.2 的完整 pipeline PASS 是用 behavioral RAM model 驗證 raw RAM port 架構。 Vivado streaming project 和 RAM IP 已建立，但 conv3_feature_ram / output_image_ram 尚未替換進 xsim 完整 pipeline 驗證。

1. Phase 目標

建立新的 RTL_sys top：sr_core_top_stream_ram_wrapper.v。 它不再例化舊的 sr_core_top，而是直接例化已驗證的 Conv1、Conv3、 PixelShuffle、OutputStage blocks，並把中間 RAM 改成對外 raw RAM port。 

TB input source
-> Conv1 Block
-> Conv1 to Conv3 register slice
-> Conv3 Block
-> conv3_feature_ram raw port
-> PixelShuffle
-> OutputStage
-> output_image_ram raw port

2. 現有問題

Phase7 / Phase8 的 sr_core_top 把 feature_mem0、 feature_mem1、feature_mem2 都寫成內部 reg[]。 這很適合逐層 debug，但不像真正 PL 架構，因為 Vivado BRAM/RAM IP 應該在同一層被明確接線。

Memory舊用途Phase9.2 決策
feature_mem0Conv1 output full feature map移除，不做 FIFO、不做 RAM
feature_mem1Conv3 output / PixelShuffle input改成 conv3_feature_ram raw port
feature_mem2PixelShuffle int8 output改成 final output_image_ram raw port

3. 為什麼 feature_mem0 不做 RAM

Conv3 是 1x1 convolution。對每個 pixel，它只需要同一個位置的 8-channel Conv1 output。 因此 Conv1 output 可以直接透過一拍 register slice 送入 Conv3，不需要保存完整 feature map。 

Conv1 q0~q7 + out_valid + x/y
-> register slice
-> Conv3 in_c0~in_c7

這個判斷跟輸入尺寸無關。即使未來變成 128x128，Conv3 仍然只讀同一 pixel 的 8 channels。 圖片越大，越不應該保留整張 feature_mem0。

4. 為什麼 feature_mem1 要變 RAM IP

PixelShuffle 目前是 write-transaction core，會按照 LR pixel 順序產生 HR address。 Conv3 output 是 PixelShuffle 的 input tensor，所以需要一個明確的 PL-side buffer： conv3_feature_ram。

本階段採用 32-bit word，而不是 8-bit word： 

conv3_feature_ram word = {c3, c2, c1, c0}
width = 32
depth = 8 * 8 = 64

這樣每個 Conv3 pixel 只需要一次 RAM write，符合 Conv3 block 一次產生 4 channel 的 parallel datapath。

5. 為什麼 feature_mem2 改成 output image RAM

sr_output_stage 是 combinational int8 to uint8 conversion。 因此 PixelShuffle 產生 HR address 後，可以直接把 int8 sub-pixels 轉成 uint8，寫入 final output RAM。 這比先存一份 int8 PixelShuffle buffer 再轉 uint8 更接近最後 PL output buffer。 

PixelShuffle wr_data_a = {c1, c0}
OutputStage converts c0/c1 to uint8
output_image_ram word = {right_uint8, left_uint8}
width = 16
depth = 16 * 16 / 2 = 128

因為 PixelShuffle 同一個 LR pixel 會同時寫 even row 和 odd row，output_image_ram 使用 true dual-port RAM 比 simple dual-port RAM 更合適。

5.1 PixelShuffle 到 OutputStage 的新接法

Phase9.2 這裡和 Phase7 的 sr_core_top 不同。Phase7 是先把 PixelShuffle 的 signed int8 結果存進 feature_mem2，再逐 pixel 送進 sr_output_stage 轉成 uint8。 Phase9.2 則把 feature_mem2 移除，讓 PixelShuffle write transaction 直接接 sr_output_stage，再寫入 final output_image_ram。 

Phase7:
PixelShuffle
-> feature_mem2, signed int8
-> sr_output_stage
-> output_mem, uint8

Phase9.2:
PixelShuffle
-> sr_output_stage, combinational int8 to uint8
-> output_image_ram, uint8

RTL 中這段看起來像新增了四個 output stage instance： 

sr_output_stage out_a_l (.in_pixel(ps_wr_data_a[7:0]),  .out_pixel_uint8(out_a_left));
sr_output_stage out_a_r (.in_pixel(ps_wr_data_a[15:8]), .out_pixel_uint8(out_a_right));
sr_output_stage out_b_l (.in_pixel(ps_wr_data_b[7:0]),  .out_pixel_uint8(out_b_left));
sr_output_stage out_b_r (.in_pixel(ps_wr_data_b[15:8]), .out_pixel_uint8(out_b_right));

原因是 pixel_shuffle_core 一次輸出兩個 packed word：

Signalbit rangePixelShuffle channelHR 位置OutputStage
ps_wr_data_a[7:0]c0even row, left pixelout_a_l
ps_wr_data_a[15:8]c1even row, right pixelout_a_r
ps_wr_data_b[7:0]c2odd row, left pixelout_b_l
ps_wr_data_b[15:8]c3odd row, right pixelout_b_r

sr_output_stage 是 combinational block，只做 output clip 與 signed int8 + 128 的 uint8 recovery。因此這裡不是新增 CNN layer，也不是額外 pipeline；它只是把原本 feature_mem2 -> output_mem 的轉換移到 output RAM write path 前面。

這樣做的好處是 final PL buffer 直接存 output_uint8 image domain。 日後若接 DMA 或 PS 讀回，就不需要再讀一份 int8 PixelShuffle buffer 後另外轉格式。

6. 參考 ref/cnn_classfication 的設計觀念

參考專案的 blk_ram_temp 不是藏在 convolution module 裡，而是由 controller 對外輸出 raw RAM port： 

ram_addr
ram_data
ram_en
ram_we

Phase9.2 採用同樣觀念：CNN datapath 負責計算，RAM IP 負責儲存，中間由 controller/top 產生 port 訊號。這比在 top 內寫 reg feature_mem[] 更接近未來 Vivado block design。

Phase9.2 分成兩層驗證。第一層先確認 RTL controller 產生的 raw RAM port 與資料流正確； 第二層建立 Vivado RAM IP prototype，保留下一步替換 behavioral RAM model 的路徑。

12. What did we learn?

Streaming 不代表所有中間資料都消失。真正的判斷標準是下一個 stage 的資料相依性。 Conv1 到 Conv3 可以 streaming，因為 Conv3 是 1x1；但 Conv3 到 PixelShuffle 需要明確的 feature/output RAM port， 因為這部分開始牽涉 HR address generation 與 output image buffer。

Phase9.2 完成後，SR Core 已經從「內部 reg array stage memory」推進到 「CNN datapath + raw RAM port」的可驗證架構；同時也建立了 Vivado RAM IP prototype。 但 Vivado RAM IP wrapper 級完整驗證留到 Phase9.3。

13. 下一步

下一步應該把 behavioral RAM model 換成 Vivado BMG RAM simulation wrapper， 確認 sr_core_top_stream_ram_wrapper 真的能和 conv3_feature_ram / output_image_ram IP 同層接線並保持 bit-exact PASS。
