Runtime-Verified Golden Flow 紀錄

主題：將 SR TFLite deployment flow 改成以 TFLite runtime 為最終 golden 的雙路徑驗證流程

回到：SR Core 最小功能設計紀錄

1. 為什麼要做這件事

原本的 sr_core_golden.py 只走 NumPy reconstructed path，也就是用我們自己重建的 int8 pipeline 去推估每一層結果： 

input
  → Conv1
  → requant + ReLU
  → Conv3
  → requant
  → PixelShuffle
  → uint8 output

這條路非常適合 RTL 設計，因為它直接對應 M0/M1、window、MAC、requantization、 PixelShuffle 這些硬體 block。

但問題是：它不一定和真正的 TFLite runtime bit-exact。 只要 rounding、saturation、zero point 修正、tensor dtype 或 intermediate tensor 選取有一點差異， reconstructed 結果就可能和真實 runtime 不同。

因此我們把 golden flow 升級成： TFLite runtime 才是最終 golden，NumPy reconstructed path 則是 RTL-oriented reference。

2. 現在的雙路徑設計

新增了 --verify-with-tflite 之後，sr_core_golden.py 會同時建立兩條路徑。

2.1 PATH A：Reconstructed Path

input_uint8
  → input_int8_for_core
  → Conv1 3x3
  → requant + ReLU
  → Conv3 1x1
  → requant
  → PixelShuffle
  → final uint8 output

這條路徑仍然保留，因為它是後續 RTL testbench 的主要對照來源。

2.2 PATH B：TFLite Runtime Path

tf.lite.Interpreter(
    model_path=...,
    experimental_preserve_all_tensors=True
)

set_tensor(input)
invoke()
get_tensor(tensor_index)

這條路徑直接從真正的 TFLite runtime 抓 intermediate tensor。 它不是硬體友善表示法，但它才是我們最終要對齊的軟體真值。

3. 這次比對哪些 Tensor

為了讓 SR core 的每一段都能逐步對齊，這次固定比對以下四個節點：

名稱 TFLite tensor index 意義
conv1_output 5 Conv1 requant + ReLU 後的 int8 feature map。
conv3_output 8 Conv3 requant 後、PixelShuffle 前的 int8 sub-pixel tensor。
pixel_shuffle_output 9 Depth-to-Space / PixelShuffle 之後的 int8 HR tensor。
final_output 12 模型最終 uint8 output。

這四個節點剛好對應你接下來會在 PL 裡逐步完成的 pipeline。

4. 每個 Tensor 會輸出什麼資訊

啟用 --verify-with-tflite 時，程式會對每個 tensor 輸出：

欄位 用途
shape 確認 runtime 與 reconstructed 的張量尺寸是否一致。
dtype 確認 int8 / uint8 型別沒有搞錯。
exact match 判斷是否 bit-exact 完全一致。
max abs diff 找出最大誤差幅度。
mismatch count 計算總共有多少 element 不一致。

這樣的目的不是只看最後 output，而是讓我們能快速定位「差異是從哪一層開始出現的」。

5. Runtime Dump 輸出目的

除了 terminal summary，程式也會輸出實際 tensor dump 到： 

model_lite/sr_core/generated/runtime_dump

每個節點都會產生： 

*_runtime.npy
*_reconstructed.npy
*_diff.npy

這樣做的目的有三個：

目的 說明
離線分析 可以用 Python / NumPy / Jupyter 直接觀察 mismatch 分布。
RTL debug 未來 PL 端輸出若錯，可以對照 reconstructed 與 runtime 看差異從哪裡開始。
流程固定化 建立 stable golden pipeline，之後每改 RTL 或 extractor 都能重新驗證。

6. 這次修改後的使用方式

一般只產生 reconstructed golden： 

python model_lite\sr_core\tools\sr_core_golden.py --height 8 --width 8 --seed 1

若要同時做 runtime verification： 

conda run -n tf_sr python model_lite\sr_core\tools\sr_core_golden.py ^
  --height 8 --width 8 --seed 1 ^
  --verify-with-tflite

這裡建議用 tf_sr 環境執行，因為 runtime 驗證需要 TensorFlow。

這表示目前的 NumPy reconstructed path，至少在這組測試條件下，已經和真正 TFLite runtime bit-exact。

8. 做這個的真正目的

做這件事不是為了讓 Python 看起來更完整，而是為了讓後續 RTL 開發有一個 可追蹤、可定位、可逐層驗證 的 golden flow。

具體來說，目的有四個：

目的 意義
確認 reconstructed model 是否可信 避免用錯誤的 Python reference 去帶 RTL 開發。
把 TFLite runtime 定位成最終 golden 之後若 reconstructed 與 runtime 有差，應以 runtime 為準。
支援逐層 debug 不是只看 final output，而是能看 conv1 / conv3 / pixel shuffle 哪一層開始偏掉。
降低 RTL 驗證風險 當 PL 輸出有誤時，可以先和 reconstructed 比，再和 runtime 比，更快定位錯誤源頭。

9. 對下一步 RTL 的幫助

現在我們有三種層次的對照資料： 

Level 1: TFLite runtime tensor
Level 2: NumPy reconstructed tensor
Level 3: 未來 RTL / simulation tensor

接下來寫 RTL 時，可以用這種方式排錯： 

RTL vs reconstructed 一致
  → 表示硬體與我們的 RTL-oriented 公式一致

reconstructed vs runtime 一致
  → 表示這套公式與 TFLite runtime 一致

所以 RTL vs runtime 也就可信

這就是把 deployment flow 改成 runtime-verified golden flow 的最大價值。
