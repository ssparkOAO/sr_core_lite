Phase8.4a ROM Timing Analysis

文件位置：model_lite/sr_core/html/phase8_4a_rom_timing_analysis.html

狀態：Analysis Only，不修改 RTL，不開始 ROM integration。

Phase8.4a 的目標是把 Vivado BMG ROM 的 1 clock read latency 分析清楚。這一步完成後，下一階段 Phase8.4b 才開始做 ROM-backed wrapper integration。

1. Behavioral Memory vs BMG ROM

Memory 類型 讀取直覺 Timing 對 Wrapper 的影響
Behavioral Memory data = mem[addr] 常被寫成 same-cycle data Wrapper 可以像查表一樣立刻取參數。
Vivado BMG ROM address 在 clock edge 被取樣 next-cycle data Wrapper 必須等待一拍，不能 address 和 data 同拍使用。 
Behavioral Memory:
cycle N:   addr -> data appears in same cycle

BMG ROM:
cycle N:   addr/en are sampled
cycle N+1: data is valid

這個差異會影響 Phase8.4b wrapper：如果直接把 Phase8.2 的 reg memory source 換成 ROM IP， weight / M0 / M1 會晚一拍到，造成 channel mapping 或 parameter mapping 錯位。

2. Conv1 Weight Access Timing

conv1_weight_rom 共有 72 筆資料，address width 是 7-bit。 mapping 仍然沿用： 

addr = output_channel * 9 + kernel_tap

co0 tap0..tap8 -> addr 0..8
co1 tap0..tap8 -> addr 9..17
...
co7 tap0..tap8 -> addr 63..71
Cycle conv1_weight_addr conv1_weight_data 說明
0 0 invalid / old 送出第一個 address。
1 1 weight[0] 收到 addr 0 的資料，同時送 addr 1。
2 2 weight[1] 收到 addr 1 的資料，同時送 addr 2。
... ... ... 每 cycle 可送下一個 address，但 data 永遠對應上一拍 address。
72 hold / don't care weight[71] 收到最後一筆資料。

取得 72 個 Conv1 weight 需要先送 72 個 address，並在最後多等一拍收尾。 若從 cycle 0 開始送 addr 0，最後一筆 weight[71] 會在 cycle 72 有效。

3. Conv3 Weight Access Timing

conv3_weight_rom 共有 32 筆資料，address width 是 5-bit。 mapping： 

addr = output_channel * 8 + input_channel

co0 ci0..ci7 -> addr 0..7
co1 ci0..ci7 -> addr 8..15
co2 ci0..ci7 -> addr 16..23
co3 ci0..ci7 -> addr 24..31
Cycle conv3_weight_addr conv3_weight_data 說明
0 0 invalid / old 送 addr 0。
1 1 weight[0] 收到 addr 0，同時送 addr 1。
2 2 weight[1] 收到 addr 1，同時送 addr 2。
32 hold / don't care weight[31] 收到最後一筆 Conv3 weight。

Conv3 weight 數量比 Conv1 少，但 timing 原則完全相同：data 對應上一拍送出的 address。

4. Conv1 M0/M1 Timing

Conv1 有 8 個 output channel，所以 conv1_m0_rom 與 conv1_m1_rom depth 都是 8，address width 是 3-bit。 

addr = output_channel

ch0 -> addr 0
ch1 -> addr 1
...
ch7 -> addr 7
Cycle conv1_m0_addr / conv1_m1_addr conv1_m0_data / conv1_m1_data 說明
0 0 invalid / old 送 channel 0 address。
1 1 M0/M1 ch0 收到 channel 0，同時送 channel 1。
... ... ... 每拍送下一個 channel。
8 hold / don't care M0/M1 ch7 收到最後一組 Conv1 requant parameter。

M0 與 M1 可以同時送同一個 channel address，下一拍同時取得該 channel 的 M0/M1。

5. Conv3 M0/M1 Timing

Conv3 有 4 個 output channel，所以 conv3_m0_rom 與 conv3_m1_rom depth 都是 4，address width 是 2-bit。

Cycle conv3_m0_addr / conv3_m1_addr conv3_m0_data / conv3_m1_data 說明
0 0 invalid / old 送 channel 0 address。
1 1 M0/M1 ch0 收到 channel 0。
2 2 M0/M1 ch1 收到 channel 1。
4 hold / don't care M0/M1 ch3 收到最後一組 Conv3 requant parameter。

6. Wrapper Timing Problem

Phase8.2 wrapper 使用 internal reg memory，概念上比較像資料已經在 wrapper 裡，可以直接展開接到 verified modules。若 Phase8.4 直接把來源換成 ROM，會遇到下列問題：

  • address 和 data 不同步：data 永遠晚 address 一拍。
  • weight 晚一拍：如果同拍把 address 當成當前 weight 使用，會拿到上一筆或 invalid data。
  • M0/M1 晚一拍：requant parameter 若錯位，輸出會大幅 mismatch。
  • start timing 不明確：如果 convolution 在 ROM preload 完成前開始，會使用到未初始化的 bridge register。
錯誤直覺：
cycle N: addr = 10, data = weight[10]

實際 BMG ROM：
cycle N:   addr = 10
cycle N+1: data = weight[10]

7. Architecture Options

Option A：Simple FSM

IDLE
  -> SEND_ADDR
  -> WAIT_ROM
  -> USE_DATA
  -> NEXT

優點：

  • 最容易理解。
  • 每筆資料都有明確 WAIT_ROM 狀態，debug 波形清楚。
  • 很適合 Phase8.4 這種 correctness-first 階段。

缺點：

  • 每筆資料至少多一個等待節拍，效率低。
  • 如果逐筆讀 72 個 weight，整體 preload 時間較長。

Option B：Prefetch Pipeline

cycle N:   send addr[i]
cycle N+1: capture data[i], send addr[i+1]
cycle N+2: capture data[i+1], send addr[i+2]

優點：

  • 填滿 pipeline 後，每 cycle 可取得一筆資料。
  • 72 個 Conv1 weight 約 73 cycles 可完成，不需要每筆都分 SEND/WAIT/USE 三個狀態。
  • 比較接近未來真正 ROM-backed architecture。

缺點：

  • 需要管理 first invalid cycle 與最後 flush cycle。
  • 需要小心 data index = addr index delayed by 1。
  • debug 比 Simple FSM 稍微複雜。

8. Recommended Solution

Phase8.4 的目標是 PASS，不是最高效能。因此建議採用保守的 preload architecture： 

ROM IP
  -> preload parameter registers
  -> existing verified sr_core_top-style ports
  -> run full pipeline
  -> compare output_uint8

實作 preload 時，可以採用 Option B 的 prefetch pipeline，因為它仍然很容易驗證，而且不會過度低效。 但 top-level 行為上要保持簡單：只有當所有 parameter registers 都載入完成後，才允許 core start。

建議：Phase8.4b 使用 prefetch preload。先把 ROM 內容載入 wrapper 內部 registers， 再沿用 Phase8.2 已驗證的 port-style top connection。這樣可以把「ROM latency 問題」和 「CNN datapath correctness」分開。

9. Expected Wrapper Timing Diagram

Cycle Addr ROM Data Valid Conv Start Conv Done
0 first parameter addr invalid 0 0 0
1 next parameter addr parameter[0] 1 0 0
2..N next addr parameter[i-1] 1 0 0
N+1 hold last parameter 1 0 0
N+2 don't care all params stored preload_done = 1 1 0
later don't care register bank used core running 0 1 when output complete

未來 sr_core_top_rom_wrapper 不應在 ROM data 還沒 valid 時啟動 convolution。 合理流程是先 preload，等 preload_done 後再啟動 core。

10. What Did We Learn?

ROM 與 behavioral memory 最大差異是 timing，不是資料內容。Phase8.3 已經證明資料內容正確， Phase8.4a 則提醒我們：BMG ROM 的資料會晚 address 一拍。

因此在 RTL integration 前必須先做 timing analysis。否則很容易遇到看似 weight mapping 錯、 requant 參數錯、channel 錯位的問題，但真正原因其實只是 ROM read latency 沒有被處理。

下一步 Phase8.4b 才開始 ROM-backed Wrapper Integration。進入 RTL 前，必須記住： data index = address index delayed by 1 cycle。
