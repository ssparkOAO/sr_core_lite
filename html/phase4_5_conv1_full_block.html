本頁記錄 Conv1 full quantized block： sr_window_3x3_cin1 → sr_conv3x3_cin1_cout8_mac → 8x sr_requantize → conv1_out_int8。

1. 本階段目標

Phase4 已經驗證 Conv1 MAC accumulator： 

input_int8_for_core
  -> window
  -> Conv1 3x3 MAC
  -> conv1_acc_int32

Phase4.5 則把 MAC output 接上每個 output channel 自己的 requant 參數， 直接比對 Conv1 的 final int8 output： 

input_int8_for_core
  -> window
  -> Conv1 3x3 MAC
  -> 8x sr_requantize
  -> conv1_out_int8

2. Conv1 Full Block 架構

sr_conv1_3x3_cin1_cout8_block
  -> sr_window_3x3_cin1
  -> sr_conv3x3_cin1_cout8_mac
  -> acc0~acc7
  -> 8x sr_requantize
  -> q0~q7

這個 block 沿用已 verified 的 window、MAC、requant module。 目前沒有加入 Conv3、PixelShuffle、AXI、DMA、BRAM IP 或 optimization pipeline。

3. Requant Dataflow

Conv1 有 8 個 output channels，所以 conv1_m0.mem 和 conv1_m1.mem 都各有 8 個值。 每個 channel 使用自己的 M0/M1： 

q0 = sr_requantize(acc0, conv1_m0[0], conv1_m1[0])
q1 = sr_requantize(acc1, conv1_m0[1], conv1_m1[1])
...
q7 = sr_requantize(acc7, conv1_m0[7], conv1_m1[7])

4. 為什麼不需要獨立 ReLU Module

No separate ReLU block after Conv1 because ReLU is fused into Conv1 and output zero_point = -128.

TFLite 中 Conv1 是 fused operator semantic：3x3 conv + requant + fused ReLU。 這個模型的 Conv1 output zero point 是 -128， 所以 quantized ReLU 的下限剛好就是 signed int8 的最小值。

sr_requantize 已經把結果 clamp 到 signed int8 範圍： 

[-128, 127]

因此對這個 verified flow 來說，不需要在 Conv1 後面再插入 standalone ReLU module。若未來遇到 output zero point 不是 -128 的 layer， 才需要重新檢查 activation clamp 的位置與語意。

5. Golden Verification Flow

Testbench 讀取：

generated/golden/input_int8_for_core.mem
generated/conv1_weight.mem
generated/conv1_m0.mem
generated/conv1_m1.mem
generated/golden/conv1_out_int8.mem

驗證流程：

1. TB 只在 in_ready 時餵 input stream
2. DUT 內部產生 3x3 window
3. DUT 內部做 Conv1 3x3 MAC
4. DUT 內部做 8-channel requant
5. TB 比對 q0~q7 與 conv1_out_int8.mem

若 mismatch，TB 會印出 pixel index、x/y、output channel、 RTL q、golden q、acc、M0、M1 和完整 3x3 window。

6. PASS 條件

mismatch count = 0
max abs diff   = 0
