本頁記錄 PixelShuffle 後的 output clip block。 這不是一般 CNN activation layer，而是 final output quantize 前的 range constraint。

1. 為什麼不是一般 ReLU Layer

TFLite graph 在 PixelShuffle 後面有 MINIMUM 和 RELU。 這裡的硬體行為不是新的 convolution activation stage，而是要把 final output quantize 前的數值限制在有效範圍內。

因此本階段稱為 output clip block， 不稱為 standalone ReLU layer。

2. TFLite MINIMUM + RELU 的意義

PixelShuffle output
  -> MINIMUM(x, 127)
  -> RELU
  -> final quantize
  -> uint8 output

依照硬體摘要，這一段的 action 是： 

clip output to valid int8 range before final output quantize

3. Signed Int8 Domain

目前 tensor domain 是 signed int8，因此有效範圍是：

[-128, 127]

Output clip 行為：

if input > 127:
    output = 127
else if input < -128:
    output = -128
else:
    output = input

4. 為什麼 Input 用較寬位元

如果 RTL input port 寫成 signed [7:0]，那 200 和 -200 不能被正確表示，進入 module 前就會 wrap。 例如 8-bit signed 無法表示 200。

因此本 block 使用較寬的 signed input 來代表「需要被 clip 的數值」， output 則固定是 signed int8： 

input  signed [15:0] in_pixel
output signed [7:0]  out_pixel

若前一級已經保證輸出是 signed int8，這個 block 會等同 passthrough。 但用較寬 input 可以正確驗證 overflow / underflow clamp 行為。

5. 為什麼不需要 Multiplier

Output clip 只是 range limiting，不做 MAC、不做 multiplier、不做 requant、 不做 line buffer、不做 BRAM、不做 state machine。 

compare
  -> mux select clamp value or original value

6. Pipeline 位置

Conv1
  -> Conv3
  -> PixelShuffle
  -> Output Clip
  -> final uint8 conversion

完成這個 block 後，下一步才會做 final uint8 format conversion。
