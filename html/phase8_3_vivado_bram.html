Phase8.3 Vivado BRAM Parameter Architecture

文件位置：model_lite/sr_core/html/phase8_3_vivado_bram.html

工作區：model_lite/sr_core/RTL_sys/phase8_3_vivado_bram/

Vivado project：model_lite/sr_core/vivado/sr_core_bram_pynqz2

Phase8.3 的目標是把 Phase8.1 / Phase8.2 已驗證過的 memory-style parameter organization， 往 Vivado Block Memory Generator IP 推進。這一階段仍然不做 AXI、DMA、PS-PL，也不修改 RTL/ verified modules。

目前已建立六個 parameter ROM prototype：Conv1 weight、Conv3 weight、Conv1 M0、Conv1 M1、Conv3 M0、Conv3 M1。

1. Phase8.3 目標

Phase8.2 已經證明：把 weight/M0/M1 放成 internal memory organization 後，仍然可以維持 end-to-end PASS。Phase8.3 接著要驗證的是：這些 parameter memory 是否能用 Vivado Block Memory Generator IP 表達，為未來 BRAM-based accelerator architecture 做準備。

本階段重點是 parameter BRAM prototype，不是 deployment。也就是先把固定參數變成可由 Vivado IP 管理的 ROM，再逐步思考未來 loader、AXI、DMA 或 PS-PL integration。

2. 目前架構

目前還沒有把 BRAM IP 接進完整 sr_core_top_mem_wrapper。這是刻意的設計： 先把 IP 初始化、address mapping、read latency、signed bit pattern 這些問題獨立驗證， 避免一開始就和整個 pipeline 混在一起。

Vivado-friendly hex / COE 會集中放在：

COE 使用 memory_initialization_radix=16，並把 signed decimal value 轉成固定寬度 two's-complement hex。這樣 Vivado BRAM 存的是正確 bit pattern，RTL 端再用 signed wire 解讀。 RTL_sys/phase8_3_vivado_bram/ 只保存 BRAM prototype RTL、Tcl、simulation，不再作為唯一資料來源。

轉檔工具使用方式請看： tool_convert_generated_to_vivado_hex.html

7. ROM 需要 enable 嗎？

不一定需要。Vivado Block Memory Generator 的 native ROM 可以不要 enable，也可以保留 ena pin。如果你的設計是每個 clock 都要依照 address 讀資料，可以把 ena 直接接成 1'b1，行為上就是「想讀就讀」。

目前 sr_param_rom_bank 保留 *_en 的原因是：

  • 未來 controller 可以明確標示哪一拍是有效 read。
  • 和 BRAM/ROM synchronous read latency 對齊時，debug waveform 比較清楚。
  • 若未來要降低不必要的 switching，可以用 enable 關掉不讀的 port。

結論：en 不是功能上必須，但保留它比較適合後續 system architecture。 如果某一版想簡化，可以在 wrapper 內把所有 *_en tie high。

9. 遇到的問題與學習

9.1 signed decimal COE 不如 two's-complement hex 穩定

generated 內原本有 signed decimal COE，但 BRAM 實際儲存的是 bit pattern。為了避免 Vivado 對負數 COE 的解析差異，本階段改產生固定 width 的 two's-complement hex COE。

9.2 Load_Init_File 和 Coe_File 要一起設定

Vivado Block Memory Generator 會立即驗證參數。如果先打開 Load_Init_File，但還沒指定 Coe_File，會出現 No COE file loaded。最後 Tcl 改成一次性 set_property -dict，同時設定兩個欄位。

9.3 BMG read latency 要在下一階段明確處理

目前 self-check 已經用 synchronous read timing 驗證。未來接到 wrapper 或 controller 時， 必須把 ROM read latency 納入 valid timing，而不是把它當成 combinational memory。

10. 目前結論

Phase8.3 已不只做 Conv1 weight，而是把目前需要的固定參數都建立成 Vivado ROM prototype。 參數初始化檔統一由 generated_vivado_hex/ 提供，避免每個 architecture phase 複製一份而產生版本落差。 下一步可以進入 Phase8.4：把 parameter ROM bank 接到 memory-oriented top wrapper， 並處理 read latency / address scheduling。
