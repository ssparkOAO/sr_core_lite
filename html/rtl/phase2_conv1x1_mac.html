本頁記錄 sr_conv1x1_cin8_cout4_mac 的目的、資料 mapping、 RTL datapath 寫法與 testbench 驗證方式。

1. 目前階段

Phase2 目前做的是 Conv3 layer 的 1x1 kernel MAC。 這裡的 Conv3 是 layer 名稱，不是 3x3 kernel。 Conv1 才是前面的 3x3 kernel layer。

這一階段只驗證：

conv1_out_int8
  -> Conv3 1x1 MAC
  -> conv3_acc_int32

目前不要加入：

M0
M1
requant
clamp
ReLU
pipeline optimization
streaming architecture

2. 數學公式

對每個 output channel：

acc[co] =
    sum_{ci=0..7}
        input[ci] * weight[co][ci]
項目 設定
Cin 8
Cout 4
activation signed int8
weight signed int8
accumulator signed int32

4. Datapath 風格

RTL 風格延續前面的 demo：明確 wire、明確 multiplier、明確 partial sum、 明確 final accumulation。這一版不是 systolic array，也沒有 PE reuse。 

input channels
  -> signed extension
  -> mul00, mul01, ..., mul37
  -> psum0a, psum0b, ...
  -> acc0, acc1, acc2, acc3

權重使用明確 port： w00~w07 屬於 output channel 0， w10~w17 屬於 output channel 1， w20~w27 屬於 output channel 2， w30~w37 屬於 output channel 3。

5. Memory Mapping

5.1 Activation input

conv1_out_int8.mem shape 是 (H,W,8)， flatten 後是 NHWC layout。每個 pixel 連續放 8 個 channel： 

[c0,c1,c2,c3,c4,c5,c6,c7]

5.2 Conv3 weight

conv3_weight.mem shape 是 [4,1,1,8]， flatten 後是 cout-major： 

w00 w01 w02 w03 w04 w05 w06 w07
w10 w11 w12 w13 w14 w15 w16 w17
w20 w21 w22 w23 w24 w25 w26 w27
w30 w31 w32 w33 w34 w35 w36 w37

5.3 Golden output ( acc = Σ fmap x weight )

conv3_acc_int32.mem shape 是 (H,W,4)。 每個 pixel 有 4 個 signed int32 accumulator： 

[acc0,acc1,acc2,acc3]

6. Testbench 驗證

Testbench 每次取一個 pixel 的 8-channel activation，送進 DUT 後比對 4 個 accumulator。

for each pixel:
    in_c0~in_c7 = conv1_out_int8[pixel]
    acc0~acc3   = DUT output
    compare with conv3_acc_int32[pixel]

目前這些 .mem 檔案內容是 signed decimal text，不是 hex。 所以 testbench 使用 $fscanf(..., "%d") 讀取。 這樣可以正確讀入負數，並保持和 golden 一致。

7. Simulation Result

mismatch count = 0
max abs diff   = 0
PASS

這代表 Conv3 layer 的 1x1 signed int8 MAC 已經和 conv3_acc_int32.mem bit-exact 一致。
