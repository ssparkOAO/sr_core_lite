sr_requantize 原理與操作紀錄

位置：model_lite/sr_core/html/rtl/sr_requantize.html

回到 RTL 進度：rtl/index.html

這頁專門記錄 sr_requantize 的目的、數學公式、 Verilog 實作重點，以及 testbench 如何拿 Conv3 layer 的 golden 做 bit-exact 驗證。

1. Module 目的

sr_requantize 是目前第一個正式 RTL module。 它不做 Conv、不做 Window、不做 PixelShuffle，也不做 pipeline。 這一關只驗證 fixed-point requantization arithmetic。

Requantization 很重要，因為 Conv MAC 會累加成 int32， 但下一層 activation 通常需要回到 int8。 所以 int32 accumulator 必須透過 M0 / M1 轉回 int8 quantized value。

命名注意：Conv1 是第一個 convolution layer，kernel 是 3x3。 Conv3 是後面的 convolution layer，kernel 是 1x1。 本頁測的是 Conv3 layer output 的 requant，不是 3x3 convolution。

2. Requantization 數學

目前公式：

q_out_tmp = acc * m0 + m1

rounded = q_out_tmp >>> 32

if q_out_tmp[31] == 1:
    rounded = rounded + 1 (tflite 是 round-half-up 對負數其實不完全對稱)

q_out = clamp(rounded, -128, 127)
訊號 寬度 意義
acc signed 32-bit Conv MAC 後的 int32 accumulator。
m0 signed 32-bit 由 TFLite quantization scale 推出來的 fixed-point multiplier。
m1 signed 64-bit zero point 與 multiplier 相關的 offset 修正項。
q_out signed 8-bit requant 後的 int8 output。

3. RTL 實作重點

3.1 signed arithmetic

RTL 裡的 input、wire、reg 都要清楚標成 signed。 如果漏掉 signed，Verilog 很容易把某些運算當成 unsigned， 負數的乘法、加法和右移就可能錯掉。

3.2 乘法寬度

目前採用比較乾淨的寫法：

wire signed [63:0] mult;

assign mult      = acc * m0;
assign q_out_tmp = mult + m1;

原因是 32-bit x 32-bit 的結果本來就是 64-bit。 之前如果寫成 128-bit 再截低 64-bit，雖然目前資料能過， 但容易讓人誤會有刻意做 128-bit 運算。

3.3 arithmetic shift

q_out_tmp >>> 32 使用 arithmetic right shift。 它會保留 sign bit，適合 signed fixed-point 數值。 如果用 logical shift >>，負數右移時可能變成錯的正數。

3.4 rounding bit ( round-half-up )

右移 32 bit 代表低 32 bit 被丟掉。 q_out_tmp[31] 是被丟掉部分的最高 bit， 目前 golden 的行為是如果這個 bit 為 1，就在 shifted 結果上加 1。

3.5 clamp 到 int8

最後結果必須限制在 signed int8 範圍： 

-128 <= q_out <= 127

5. Testbench 使用資料

目前使用 Conv3 layer 的 golden，因為 Conv3 是 1x1 kernel， 而且 Conv3 output 後面沒有 ReLU，最單純： 

conv3_acc_int32
  -> requant
  -> conv3_out_int8
資料 檔案
acc input generated/golden/conv3_acc_int32.mem
m0 generated/conv3_m0.mem
m1 generated/conv3_m1.mem
golden output generated/golden/conv3_out_int8.mem

原本需求寫使用 $readmemh，但目前這些 .mem 實際內容是 signed decimal text，例如 -8758、60421536。 所以 testbench 使用 $fscanf(..., "%d") 讀取， 這樣才能正確讀入負數與十進位資料。

6. Testbench 流程

1. 開啟 conv3_acc_int32.mem
2. 開啟 conv3_m0.mem
3. 開啟 conv3_m1.mem
4. 開啟 conv3_out_int8.mem
5. 讀入所有 golden / parameter
6. 逐筆送入 acc, m0, m1
7. 等 combinational output 穩定
8. 比對 RTL q_out 與 golden
9. 統計 mismatch count 與 max abs diff

Conv3 layer 的 Cout = 4，所以每個 output index 對應的 channel 是 0, 1, 2, 3 循環： 

ch = i % COUT;

7. Simulation 輸出

simulation 結束時印出：

mismatch count
max abs diff

若 mismatch，會印出：

index
acc
m0
m1
golden
rtl_out

目前結果：

mismatch count = 0
max abs diff   = 0
PASS

8. 這一關的意義

這一關 PASS 代表 RTL 的 requantization arithmetic 已經與 runtime-verified TFLite golden 完全一致。 接下來才適合把這個 module 接到 Conv MAC、ReLU、PixelShuffle 等 dataflow module。
