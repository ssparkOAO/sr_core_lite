Phase11 Wrapper 架構比較：控制型 IP 與 DMA Streaming IP

位置：model_lite/sr_core/PS_PL/html/phase11/wrapper_architecture_comparison.html

這篇記錄為什麼 ref/cnn_classfication 的專案看起來是 myip_v1_0 控制 NN_1，而目前 SR accelerator 則是把 sr_core_axis_wrapper 外面再包成 sr_accel_axis_ip_top， 並接到 AXI DMA。

1. 問題背景

參考專案 ref/cnn_classfication 的 Block Design 裡，可以看到 myip_v1_0 和 NN_1 是分開的。myip_v1_0 連到 PS/AXI，並控制 NN_1 的 start、RAM address、RAM enable、 write enable 與 output result。

這會讓人疑惑：為什麼它不是直接把 CNN core 包成 AXI？為什麼還需要另外一個 IP 去控制 CNN 電路？

2. ref 專案的架構：控制型 Wrapper

PS / AXI control
        |
        v
myip_v1_0
        |
        +-> start_flag / end_flag
        +-> RAM address / RAM enable / RAM write enable
        +-> result readback
        |
        v
NN_1 CNN datapath

NN_1 本身不是 AXI IP。它比較像一顆裸 CNN core，對外露出的訊號是 start_flag、end_flag、RAM address、RAM data、 RAM enable、RAM write enable 等硬體控制訊號。

因此 myip_v1_0 的角色是把 PS 端看得懂的 AXI control 轉成 NN_1 需要的本地控制訊號。它不是 CNN arithmetic 本身，而是 「PS/AXI 控制外殼」。

3. 目前 SR 專案的架構：DMA Streaming Wrapper

PS DDR input buffer
        |
        v
AXI DMA MM2S
        |
        v
sr_accel_axis_ip_top
        |
        v
sr_core_axis_wrapper
        |
        v
Frozen SR Core
        |
        v
AXI DMA S2MM
        |
        v
PS DDR output buffer

SR accelerator 的輸入與輸出都是影像資料，資料量遠比分類結果大。若用 AXI-Lite register 一筆一筆搬 pixel，會非常沒效率。因此目前選擇 AXI DMA + AXI4-Stream： PS 端把 input buffer 放在 DDR，DMA 以 stream 形式送入 PL；SR core 完成後， output stream 再由 DMA 寫回 DDR。

sr_accel_axis_ip_top 是給 Vivado IP Packager 和 AXI DMA 看的乾淨外殼。 它把外部介面整理成標準的 S_AXIS / M_AXIS、aclk、 aresetn，內部再接到已驗證通過的 sr_core_axis_wrapper。

4. 兩種 Wrapper 的差異

項目 ref/cnn_classfication 目前 SR accelerator
外層 IP myip_v1_0 sr_accel_axis_ip_top
內部 CNN core NN_1 sr_core_axis_wrapper + frozen SR core
PS 溝通方式 AXI-Lite / AXI control register AXI DMA stream
主要資料搬運 外層 IP 控 RAM / start / result DMA MM2S 輸入，DMA S2MM 輸出
適合任務 分類結果少、控制訊號多 影像輸入輸出大、pixel stream 多
設計重點 讓 PS 控制 CNN core 與 RAM 讓 DMA 高效率搬整張影像

5. 為什麼不是直接把 Frozen Core 包成 AXI？

Frozen SR core 是已驗證的 compute datapath。它的目標是維持功能正確，不要在系統整合時 反覆修改。若直接把 AXI protocol 寫進 frozen core，會把 CNN arithmetic、stream protocol、 DMA timing、reset、packaging 規則混在一起，debug 會變困難。

因此目前採用分層：Frozen core 負責 SR 計算；sr_core_axis_wrapper 負責 AXI stream frame 行為；sr_accel_axis_ip_top 負責 Vivado IP packaging 友善介面。

6. 這和 ref 專案其實是同一種思想

表面上看，ref 是 myip_v1_0 控 NN_1，目前 SR 是 sr_accel_axis_ip_top 包 sr_core_axis_wrapper。 但本質一樣：都沒有把 CNN datapath 直接暴露給 PS，而是在 CNN core 外面加一層 system-facing wrapper。 

ref:
PS-friendly control IP
-> CNN core

SR:
DMA-friendly streaming IP
-> SR core

差異只在於資料搬運方式。分類專案 output 很少，所以 control/register-centric 架構合理； SR 專案要搬 128x128 input 與 256x256 output，所以 DMA streaming 架構更合理。

7. 目前結論

sr_accel_axis_ip_top 不是多餘的一層。它讓 Vivado IP Packager 看到乾淨的 AXI4-Stream 介面，也保護已驗證的 SR core 不被 PS/PL integration 細節污染。

未來若要加入 AXI-Lite control/status，例如 start、done、error、image size、interrupt， 可以再新增一層 control wrapper，概念就會更接近 ref 的 myip_v1_0。
