25 MHz timing pass 證明目前主要問題不是 AXI DMA、SmartConnect 或 IP packaging， 而是 clean-stream SR core 的 Conv1 datapath 在 100 MHz 下太長。降頻後，每個 cycle 有 40 ns，足以容納目前約 35 ns 的 worst path。

這條 worst path 從 Conv1 window register 出發，穿過 Conv1 MAC / requant 相關組合邏輯， 最後到 Conv1 -> Conv3 streaming slice 的輸出暫存器。也就是說，主要問題不是 AXI DMA 或 SmartConnect，而是 SR core 內部 Conv1 datapath 太長。

5. 為什麼 100 MHz 會失敗

目前 Phase9.6 clean-stream baseline 是以功能正確與 hierarchy readability 為優先， 並沒有為 100 MHz timing closure 做 pipeline。Conv1 3x3 Cout=8 的 MAC、requant、 control selection 與 Conv1 -> Conv3 slice 之間形成太長的 combinational path。

目前 100 MHz 給每個 clock cycle 10 ns，但 worst path data delay 約 35 ns。 這代表不是簡單加 implementation strategy 就能完全解決，核心 datapath 需要切 pipeline， 或先降低 PL clock 作為 bring-up 測試。

6. DRC / Methodology 訊息

  • Route status：routing errors = 0。
  • DRC 主要 warning 包含 DSP input/output pipeline 建議，例如 DPIP-1、DPOP-1、DPOP-2。
  • Methodology DRC 顯示 TIMING-16 large setup violation，與 timing summary 的 setup failure 一致。

DSP pipeline warning 很重要，因為 Conv1 MAC 使用 DSP48E1，但目前未充分使用 DSP input/output register。 這會讓 multiplier / adder tree 更難在 100 MHz 下收斂。

7. 下一步建議

方案 用途 優缺點
方案 A：先降 FCLK 快速確認板上 DMA / Python 流程能不能跑。 已驗證 25 MHz timing pass。適合 Phase11.2 先做 DMA smoke test。
方案 B：Conv1 pipeline 在 Conv1 MAC partial sum、requant、Conv1->Conv3 slice 之間切暫存器。 正規解法，但要重新做 RTL simulation、AXIS wrapper simulation、Vivado implementation。
方案 C：使用 DSP pipeline register 讓乘法與加法路徑更容易 timing closure。 需要調整 MAC RTL 或 inference style，仍需重新驗證 bit-exact 行為。
方案 D：多 cycle path 若架構允許某些計算跨多 cycle，可用約束反映真實 timing。 目前 Conv1 -> Conv3 是 streaming valid path，不應直接亂加 multicycle，除非 RTL control 明確支援。

8. 目前結論

Phase11.1 可以記錄為：Validate Design PASS、Synthesis PASS、Run Implementation PASS、 Generate Bitstream PASS。Timing 狀態要分開看：100 MHz FAIL，25 MHz PASS。 目前可用 25 MHz bitstream 進入 Phase11.2 DMA smoke test。

若目標是正式可部署 accelerator，後續仍應回到 clean-stream core 做 Conv1 pipeline / DSP pipeline timing refactor，把可用頻率往 100 MHz 拉回來。
